Describing Valid Signing Scenarios

The Electronic Signatures feature accommodates various signing scenarios for different business needs. The application requires only a signer name and password to fulfill signature requirements, but provides several other options when a signer cannot be authenticated or is unavailable. Valid signing scenarios include:

• Signer only

The signer completes the Signer and Password fields to add a signature to the requirement.

• Signer and cosigner, when the signer cannot be authenticated

The signer completes the Signer field, and a cosigner completes the Cosigner, Cosigner Password, and Cosign Reason fields to add a signature to the requirement. The signer must leave the Signer Password blank in this scenario. Although the cosigner username and password may be valid, the application does not allow you to add a signature with an invalid signer password. Both signatures will be viewable in the audit trail.

• Signer and cosigner, when the signer has a valid username and password but has not been certified to sign for a requirement

The signer completes the Signer field and Signer Password field, and a cosigner completes the Cosigner, Cosigner Password, and Cosign Reason fields to add a signature to the requirement. While the application will accept only the signer name, adding the cosigner name may be necessary for regulatory and auditing purposes. Both signatures will be viewable in the audit trail.

• Cosigner only, when the signer is unavailable

The cosigner completes the Cosigner, Cosigner Password, and Cosign Reason fields to add a signature to the requirement. A signer username and password are not required.

Important: This scenario is valid when capturing signatures for a single-container transaction and for a multi-container transaction in individual mode. When capturing signatures for a multi-transaction in batch mode, the application requires a Signer.
Note: The cosigner fields are available only if a cosigner role was specified for the signature requirement in the ESig requirement modeling object.
