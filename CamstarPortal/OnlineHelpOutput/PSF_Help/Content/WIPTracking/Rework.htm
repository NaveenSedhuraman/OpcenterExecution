Rework Containers

The Rework transaction indicates processing is complete at a step but the intended results were not achieved. The Rework transaction allows you to move a container from the current workflow step to another workflow step so that certain procedures can be repeated or additional work can be completed. Rework is a type of move transaction that has additional capabilities.

When Using Rework

All display fields are empty on the Rework page until a container is selected. Display fields are populated automatically on the Rework pop-up. Refer to "Using Transaction Pages and Pop-ups" for information on performing a transaction from a page or pop-up.

The ability to rework a container is specific to an operation. Rework reasons must be defined and associated with an operation (in Modeling) before they can be used with the Rework transaction. A rework reason explains why a container is being reworked and may also indicate where the container will go after being reworked. You can determine all the extra work that was performed related to a particular problem from a cost accounting viewpoint. Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guidefor information on rework reasons.

Important: The application prevents you from executing the rework transaction if you typed a rework reason in the Rework Reason field that is not associated with the container's current operation. The application displays an error message when you click the Rework button if you typed an invalid rework reason.

The Rework transaction is available only for those containers at steps (specs) for which a rework path has been defined in the workflow. If the product is assigned a workflow that does not have an associated rework path, then no paths will be displayed in the rework Path field.

When Using the Rework Pop-up

The application displays a default value in the Resource field when one of these conditions is true:

• A move in was required for the container's current step and the user moving in the container specified a resource.

Or

• A resource was specified when the container was started and the Rework is the first transaction (specifying a resource) that you are performing after starting the container.

You can select a different resource, if necessary.

Rework Page

You can access the Rework page from the Container menu. The Rework page allows you to select a container for rework and assign a reason for the rework. It also provides the ability to assign a rework path.

Note: The Rework page may also be available as a pop-up if the container hold transaction has been configured as an available action on another page (for example, Container Search).

Rework Page and Pop-up Field Definitions

This table defines the fields on the Rework page and Rework pop-up.

Resource that processed the container at the current step. You can select a different resource if the application displays one by default. Specifying the resource used at this step before moving the container provides traceability.

How to Move a Container to the Rework Step

Follow these steps to move a container to the rework step:

1. Open the Rework page from the Container menu. The Rework page appears.
2. Select or enter a container in the Container field. The application updates the container header and selection fields to display information for the specified container. Values appear only if they were entered for the container.
Note: The application populates the Container field and the container header automatically when you access the Rework pop-up from another shop floor page.
3. Select a Rework Reason.
4. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
5. Click the Rework or Submit on the command bar to move the container to the specified step. A status message appears confirming that the container has been moved.
Important: The application prevents you from executing the rework transaction if you entered a rework reason in the Rework Reason field that is not associated with the container's current operation. The application displays an error message when you click the Rework button if you entered an invalid rework reason.
