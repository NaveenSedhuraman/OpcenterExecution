Describing Electronic Signatures

According to the Food and Drug Administration (FDA), Electronic Signatures are intended to be the equivalent of handwritten signatures, initials, and other general signings required by predicate rules. Part 11 signatures include electronic signatures that are used, for example, to document that certain events or actions occurred in accordance with the predicate rule (for example, approved, reviewed, and verified).

For example, you can define an electronic signature requirement that automatically deploys an ESignature pop-up during the Move Non-std transaction. When you establish this requirement, a shop floor user cannot execute the Move Non-std transaction until a signer with appropriate authority has provided a signature approving the transaction.

Each captured signature is tracked and stored in transaction history. Each entry contains the signature requirements and how they were met, along with the local time and GMT. Consequently, you can generate a complete audit trail and device history record (DHR) from this data.

When Capturing Electronic Signatures

An employee can sign off only once on an ESig requirement regardless of the number of meanings assigned to the requirement.

The ESignature pop-up appears when you click a transaction button if:

• An electronic signature requirement has been configured for the transaction in Modeling.

Or

• A transaction attempts to stop a process timer that has an ESig Requirement and is outside of its limits.

When launching the ESignature pop-up from a shop floor transaction page or pop-up for a single container, you can add signatures for that container only. The signature requirements displayed in the Electronic Signature Entry grid are from the electronic signature requirement associated with the transaction at the container's current spec and any relevant process timers.

When launching the ESignature pop-up from a multi-container shop floor transaction page or pop-up, you can add signatures for multiple containers individually or as a batch. The signature requirements displayed may be from multiple electronic signature requirements. Refer to "" for information on capturing electronic signatures for multiple containers.

Note: Multi-container transaction pop-ups are launched from the Container Search Page while multi-container transaction pages are launched from the Container menu (if the menu has been configured with those pages).

The ESignature pop-up appearing causes the application to:

• Expand the parent and child grids of all signature requirements by default. The application displays empty child rows for every parent row. The number of child rows displayed depends on the count value defined in the electronic signature requirement. For example, if the Count is 3, the application displays 3 blank child rows for that parent.
Note: The parent and child grids remain expanded unless you collapse them manually.
• Select the first blank child row for the first unfulfilled signature requirement. If all signatures for all requirements have been added, the application selects the first child row for the first parent. 
• Place the cursor in the Signer field.
Note: The application adds scrolling to the Electronic Signature Entry grid if the expanded signature requirements exceed the height of the grid.

The meaning displayed in each parent signature requirement row identifies the purpose of the signature. Additionally, each parent row displays this information:

• Count - the number of signatures required
• Role - the role required to sign
• Cosigner Role - the role that can cosign for the meaning
Note: The application displays a cosigner role only if one was specified for the meaning in the electronic signature requirement.

The application adds the signer information to the grid when you add a signature to the selected requirement. The application changes the background color of the parent row to green when all signatures have been added for a meaning.

When Capturing Electronic Signatures for Multiple Containers

The multi-container electronic signature feature allows you to apply signatures to more than one container at the same time when executing a transaction on those containers from a multi-container transaction page or pop-up. You can apply a valid user signature to multiple containers for a common meaning regardless of where the containers are in a workflow and the electronic signature requirement associated at their current spec. The containers can be at different specs, with different electronic signature requirements associated with the transaction you are trying to perform.

Note: Remember, an employee can sign off only once on an ESig requirement regardless of the number of meanings assigned to the requirement.

Additionally, the multi-container electronic signature feature provides two modes:

• Batch, the default, which applies the user's role, username, and password to all meanings satisfied by that information
• Individual, which allows a user to add a signature for one meaning at a time

The application adds the following to the interface when the ESignature pop-up is launched from a multi-container transaction page or pop-up:

• Containers link

The Containers link indicates the number of containers to which the signature will be added. Click the link to view a list of the containers.

• Perform batch signoff for all satisfied meanings (no row selection required) radio button

This option, batch mode, is selected by default and all the parent rows in the Electronic Signature Entry grid are expanded. The application updates the selections in the Signer Role field to include the roles specified in the signature requirements listed in the Electronic Signature Entry grid.

• Perform individual signoff for each meaning (row selection required) radio button

Selecting this option, individual mode, disables the Signer Role field and the application populates the field with the role value of the selected parent row.

• Signer Role field

The availability of this field depends on the mode selected. The field is enabled in batch mode and disabled in individual mode.

When adding a signature to multiple containers, the application removes all duplicate signature requirements from the Electronic Signature Entry grid. So, a single meaning may apply to multiple containers. Clicking the Containers link in the upper right corner of the ESignature pop-up displays all the containers to which the signature will be added.

When Capturing Signatures for Multiple Meanings in Batch Mode

The Perform Batch Sign off for all satisfied meanings (no row selection required) mode allows you to add signatures for multiple meanings at once. This option is selected and the Signer Role field is enabled by default when the ESignature pop-up appears. The application updates the selections in the Signer Role field based on the roles for the meanings in the Electronic Signature Entry grid.

When in batch mode, all meanings in the Electronic Signature Entry grid are expanded by default. When the user selects his role, enters his user ID and password, and clicks Add Signature, the application:

• Automatically determines which meanings are satisfied with that role, username, and password combination
• Adds the signatures to the relevant signature requirements
• Highlights the added signatures in the expanded parent rows
• Adds a new child row to the parent row if there are remaining signatures to capture for the signature requirement

The Signer Role and Signer fields are required in batch mode, but the Signer Password field is optional.

When Capturing Signatures for an Individual Meaning in Batch Mode

Selecting the Perform individual signoff for each meaning (row selection required) mode allows you to add signatures to one meaning at a time. When you select this option, the application disables the Signer Role field and populates it with the role for the selected meaning.

The functionality of capturing signatures in individual mode is identical to capturing signatures for a single-container transaction. The difference is that when adding a signature to a meaning, you may be adding the signature for multiple containers at once.

The Signer and Signer Password fields are optional in individual mode.

Batch Mode Example, Signer Only

You can perform a batch signoff so that the user's signature is applied to every meaning that his role, username, and password satisfy. You must have the Perform batch signoff for all satisfied meanings (no row selection required) option selected on the ESignature pop-up to apply the signature in a batch mode.

Note these results if a user with the Operator role applies his signature in batch mode:

• Container 001 at Spec A: The Verify work performed meaning and the electronic signature requirement are satisfied. The application can complete the transaction for the container.
• Container 1AB at Spec B:
• One signature has been applied to the Verify work performed meaning, but the meaning requires a second signature. The meaning is not satisfied, so the signature requirement is not satisfied.
• The Product ready to ship meaning has not been satisfied.
• The user cannot submit the signatures until all signatures for all meanings have been satisfied.
• Container Z12 at Spec C: The Verify work performed meaning is satisfied, but the electronic signature requirement is not. The user cannot submit the signatures until all signatures for all meanings have been satisfied.

Individual Mode Example, Signer Only

You can perform a batch signoff so that the user's signature is applied to a single meaning. You must have the Perform individual signoff for each meaning (row selection required) option selected on the ESignature pop-up to apply the signature to individual meanings.

Note these results if a user with the Operator role applies his signature in individual mode:

• Container 001 at Spec A: The Verify work performed meaning and the electronic signature requirement are satisfied. The application can complete the transaction for the container.
• Container 1AB at Spec B: One signature has been applied to the Verify Work Performed meaning, but the meaning requires a second signature. Neither the meaning nor the signature requirement are satisfied. The user cannot submit the signatures until all signatures for all meanings have been satisfied.
• Container Z12 at Spec C: The Verify work performed meaning is satisfied, but the electronic signature requirement is not. The user cannot submit the signatures until all signatures for all meanings have been satisfied.
