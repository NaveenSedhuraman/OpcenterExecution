Two-Level Start

The Start - Two Level page allows you to start two different levels of containers. Quantities of single-level containers are defined, while quantities of multi-level containers are derived (or rolled up) from the quantities of the children containers.

Many companies track smaller (child) containers within a larger (parent) container. The group of parent and child containers are processed and tracked together. Therefore, the Start - Two Level page allows you to start parent and child containers at the same time.

Multi-level containers are started in a multilevel structure of two or more hierarchical levels; the containers in the structure have parent-child relationships.

If containers are being processed in the same way, starting them as a multi-level structure provides the benefit of entering subsequent move-type transactions at the top-level container only while the transaction is carried out through all the child containers.

Multi-Level Tracking in Manufacturing

In Manufacturing, the processing of parent and child containers is accomplished using multi-level tracking. Multi-level tracking:

• Provides tracking of multiple levels of containers.
• Provides the ability to start a group of parent and child containers and group/ungroup existing containers.
• Supports performing Manufacturing transactions against container groups.

When Using Two-Level Start

Container names must be unique regardless of container level.

You can enforce a hierarchy of valid level relationships. For example, you can allow a batch to have cassettes as children, but disallow a cassette to have batches as children.

When starting a group of containers, the children inherit the attributes defined at the parent level. Children’s attributes can be modified later using the Container Maintenance transaction.

Factory is not a required field; however, a factory must either be assigned to the employee starting the container, or be specified on the Start - Two Level page when you start the container. Otherwise, other transactions such as Move will not process the container.

Note: The application displays a value in the Factory field automatically if a factory was specified for the logged in employee. You can select a different factory, if necessary.

All display fields are empty on the Start - Two Level page until a container is selected. Display fields are populated automatically on the Child Container Info pop-up. Refer to "Using Transaction Pages and Pop-ups" for information on performing a transaction from a page or pop-up.

Displaying the Child Container Info pop-up causes the application to populate the fields on the pop-up based on current child container values made at time the child container was added to the Child Container grid. Container Name and Level cannot be modified when using auto numbering for child names.

The Clear All button clears all of the fields on the Start - Two Level page. However, on the Child Container Info pop-up, the Clear All button resets the fields on the pop-up to the values that were set when it was opened initially.

Click Update on the Child Container Info pop-up to save any changes, close the pop-up, and return to the Start - Two Level page.

Click the Close button to close the Child Container Info pop-up without saving any changes.

When Naming Parent and Child Containers

You must specify a unique container name when creating new parent containers. You can either enter a container name manually or have the application generate a container name automatically. The application considers closed containers when validating a unique name.

The application uses the parent container name to name child containers regardless of the method used to name the parent. The default method is to append – 01, - 02, etc. to the parent name for each child container. However, additional naming options are available. Refer to "When Generating Child Container Names” for information.

Note: The application displays the parent container in the success message when the transaction executes successfully. The application does not include child container names in the success message.

When Generating a Parent Container Name

You cannot change the method for generating the parent name after creating one or more child containers. The application clears the Child Containers grid and displays a warning message if you change the method after creating child containers.

Select the Generate Names Automatically check box to have the application generate the parent container name using defined numbering rules. The application clears and disables the Container field when you select this check box.

The application uses the numbering rules associated with the parent container to generate the container name. Numbering rules may be specified for multiple modeling object instances associated with the container. Numbering rules follow this default precedence:

1. Container Level
2. MfgOrder
3. Product
4. Product Family
5. Factory

Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information on defining numbering rules.

When Generating Child Container Names

You must associate a numbering rule with the container level you want to use when generating child container names automatically. The application displays a warning message when:

• You select a container level that does not have an associated numbering rule,
• You select the Generate Child Names Automatically check box, and
• Click Add.

The application uses multiple criteria to determine child container names. There are four possible scenarios:

• When both the Generate Names Automatically and Generate Child Names Automatically check boxes are cleared (the default setting), the application assigns each child container the name specified in the Container field and appends -01, -02, etc. to the end.

Example:

MyContainer – 01

MyContainer – 02

MyContainer – 03

• When the Generate Names Automatically check box is selected and the Generate Child Names Automatically check box is cleared, the application assigns each child container name the parent container name generated by associated numbering rule and appends -01, -02, etc. to the end.

Example:

c_0001 – 01

c_0001 – 02

c_0001 – 03

• When the Generate Names Automatically check box is cleared and the Generate Child Names Automatically check box is selected, the application assigns each child container the name specified in the Container field and appends the numbering rule associated with the child container Level to the end.

Example:

MyContainersCC0001

MyContainersCC0002

MyContainersCC0003

• When both the Generate Names Automatically and Generate Child Names Automatically check boxes are selected, the application assigns each child container a concatenation of the generated parent and child numbering rules.

Example:

c_0002CC0003

c_0002CC0004

c_0002CC0005

Start - Two Level Page

You can access the Start - Two Level page by selecting Start - Two Level from the Container menu. The Start - Two Level page allows you to start a single container or a parent container with associated child containers.

Child Container Info Pop-up

The Child Container Info pop-up allows you to update information for child containers entered on the Start - Two Level page. Selecting a child container in the Child Containers grid and clicking the Edit selected row icon displays the Child Container Info pop-up.

How to Start Multiple Containers

Follow these steps to start multiple containers:

1. Open the Start - Two Level page from the Container menu. The Start - Two Level page appears.
2. Enter a name for the parent container in the Container Name field.

Or

Select the Generate Names Automatically check box.

3. Select a Level. The selected level appears in the Level field.
4. Select an Owner. The selected owner appears in the Owner field.
5. Select a Start Reason. The selected reason appears in the Start Reason field.
6. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
7. Perform the "How to Start Child Containers" procedure to start new child containers.
8. Click Start or Submit on the command bar. A status message appears confirming that the containers have been started.
Note: A warning message appears when Generate Names Automatically is selected but there is no auto numbering rule defined.

How to Start Child Containers

Follow these steps to start child containers:

1. Perform the "How to Start Multiple Containers" procedure.
2. Select a Level for the child containers.
3. Select the Generate Child Names Automatically check box if you want the child container names to be automatically generated.
4. Enter a Qty for the child containers.
5. Enter a UOM for the quantity.
6. Enter the number of child containers to create in the Child Count field. The application enables the Add button.
7. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
8. Click Add. The child containers are added to the Child Containers grid.
9. Repeat steps 2-8 to start additional child containers or steps, or click Add new row in the grid to display the Child Container Info pop-up.
Note: When you select Generate Child Names Automatically for the first child container, you must select the same for all subsequent child containers.
Note: Refer to the field definitions table for information on the Child Container Info pop-up.

How to Edit Child Containers

Follow these steps to edit child containers:

1. Perform the "How to Start Child Containers" procedure.
2. Select the child container you want to edit.
3. Click Edit selected row in the grid. The Child Container Info pop-up appears.
4. Enter or change information as necessary.
Note: Container Name and Level cannot be modified when using auto numbering for the child names.
5. Click Update. The application saves your changes and closes the pop-up.
