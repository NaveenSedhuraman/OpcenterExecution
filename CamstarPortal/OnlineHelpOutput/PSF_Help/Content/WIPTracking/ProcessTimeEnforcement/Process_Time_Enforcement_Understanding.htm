Understanding Process Time Enforcement

Process Time Enforcement enables you to associate process timers to containers. Process timers determine the actions to take if an operator or piece of equipment tries to perform a transaction on a container before a minimum time is met or after the maximum time allowed has passed.

The application performs a validation when a transaction is executed to ensure there are no timers out of their defined bounds. The action defined in the timer must be performed when a transaction is executed on that container if a container's timer is out of bounds.

Configuring Process Time Enforcement in Modeling

A system administrator must perform these tasks to set up process time enforcement:

• Define one or more process timers based on your business needs.
• Add one or more process timers to a spec. Process timers may only be associated with the transactions Move or Move In by default.
• Add one or more process timers to a task. Process timers can only be added to Move or Move In transaction tasks by default.

Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information.

About Process Timers

The only transactions that can trigger a timer are:

• Move
• Move In
• Execute Task
Note: Process timers do not take into account the number of iterations that are defined for a task. The process timer will only execute on the first execution of the task.

The Move and Move In transactions trigger timers for a container:

• At a spec if one or more process timers have been configured on the spec.

Or

• In an electronic procedure if one or more process timers have been configured on a transaction type task and the Move or Move In transactions are specified for that task.

The ExecuteTask transaction in electronic procedures triggers timers for a container if one or more process timers have been configured on the task. You can add process timers to all task types: instruction, computation, and transaction.

The container and timer information history is stored and is available in the manufacturing audit trail.

The application checks for active process timers when the Close transaction is used to close a container. Any timers present are stopped and the Close transaction in the manufacturing audit trail for the container displays the Process Timer History detail for any timers that are stopped.

Viewing Process Timers

Multiple timers can be triggered for a container. The most critical timer is the timer that appears in the Timers field and in the first row of the Active Timers flyout. Timers with a value in the Time to Max field are considered most critical. If there is more than one timer with a value in the Time to Max field then the timer with the least time remaining is the most critical timer. If there are no timers with a value in the Time to Max field then the timer with the least time remaining in the Time to Min field is the most critical timer.

Any available timers are displayed in these locations in the application:

• Container Information - The Timers field appears in the container information. The most critical timer appears in the field. Clicking the triangle next to the Timers field label opens the Active Timers flyout and shows all active timers. Refer to "Container Information" for information.
• Operational View page - The Timers field on the Operational View page appears in the In Queue Containers grid and the In Process Containers grid. The most critical timer appears in the field and chevrons that activate the Active Timers flyout when clicked. All other active timers are displayed on the Active Timers flyout. Refer to "Operational View" for information.
• Process Timer Search page - This page enables you to search for timers, sort the results, view timers, and stop timers. Refer to "Process Timer Search" for information.
Important: Unexpected behavior may occur if there is a discrepancy between the server clock and the local client clock. The time displayed is based on the local client clock, but the actual time is controlled by the server clock. For example, a timer may display the minimum time has been reached, but the server may be several seconds behind. Executing a transaction may produce an error until the server clock catches up.

How Process Timers Work

Required Timer Information

The system administrator should include the following information when defining a timer:

• Timer name
• Minimum end time, maximum end time, or both

Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information.

Optional Timer Information

The system administrator can associate colors with each minimum end time and maximum end time when setting up the process timer in Modeling. The application displays the color in the timer field anywhere the timer is visible if the minimum end time or maximum end time has been reached. Timers are visible on the container information, grids on the Operational View page, and the Process Timer Search page.

Process timers can have a minimum warning time or a maximum warning time. Colors can be assigned to warning times in Modeling. This sets a warning period for the container alerting you that the minimum end time or maximum end time is approaching. When the minimum warning time or maximum warning time is reached, the background color of the timer field or grid row changes, indicating that the container is in the warning period.

Multiple Timers on a Container

You can start the same timer an unlimited number of times on the same container. The instances of those timers may run concurrently. The application stops the timer that has been running the longest when an end timer transaction for the timer is executed. The remaining instances of the timer continue to run.

The application prevents duplicate timers from running. A duplicate timer is a timer of the same name, on the same container, started by the same transaction, at the same step in the workflow. This scenario can occur if there is rework in the workflow. The application stops the running timer and starts a new instance of that timer.

Timer Less Than Min or Greater Than Max

The system administrator can specify the application's behavior when a transaction attempts to execute and the timer has not reached the min time or exceeds the max time.

• Do not allow the end transaction and any disposition actions to execute:
• The timer continues to run.
• The end transaction is allowed once the min time has been reached.
• You must manually manipulate the container (execute a Move Non-Standard transaction) and stop the timer with the Stop Timer button on the Timer Search Page if the max time has been exceeded.
• Allow the end transaction and any disposition actions to execute:
• The timer stops.
• The end transaction and any disposition actions execute.
Note: The system administrator may configure a timer to require a confirmation dialog to display before executing the end transaction and disposition actions. You can select Yes in the dialog to execute the end transaction and disposition actions or No to return to the previous screen.
• Allow the end transaction to execute with an electronic signature and do not execute any disposition actions.
• An ESignature pop-up displays before the transaction is completed.
• The end transaction executes if the ESignature requirement is properly satisfied.
• The end transaction does not execute and the timer does not stop if the ESignature requirement is not properly satisfied.
Note: All ESignatures display on a single pop-up when ESignatures are required for multiple timers on a transaction. If the pop-up is closed without satisfying all ESignature requirements, the transaction does not execute and the timer does not stop.

Disposition Actions

The system administrator can specify disposition actions the application performs when the end transaction executes out of bounds. The system administrator can choose from the following disposition actions which are executed in this order:

1. Business Rule
2. Production Event
3. Rework
4. Move Non-Standard
5. Hold
Note: The configuration of the container's workflow does not impact the order of the disposition actions.

The end transaction executes successfully and no other actions execute if the administrator does not specify disposition actions and the timer is set up to allow end transactions and disposition actions. Subsequent disposition actions are not executed if more than one disposition action is specified and one fails. For example, both a production event and rework are specified. The rework transaction is not executed if the production event fails. A successful disposition action will not be rolled back if a subsequent disposition action fails. No disposition actions are executed if the timer is stopped using the Stop Timer button on the Process Timer Search page. You can specify to allow an end transaction to execute and stop a timer outside of its limits with an electronic signature. Disposition actions are not executed with this option.

Note: Production Event is a function of Quality, a licensed feature. A Production Event disposition action fails if your organization is not licensed for Quality. All subsequent disposition actions will not execute. The application does not generate completion or error messages for disposition actions.
