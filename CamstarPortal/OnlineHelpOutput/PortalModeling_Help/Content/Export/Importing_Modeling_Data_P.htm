Importing Modeling Data

The Portal in Opcenter EX MDD and Opcenter EX CR provides the ability to upload and import exported modeling object instances from one installation to another. You can perform a partial import by specifying selected instances you do not want to include in the import rather than selecting the entire export file for import.

Note: The default settings for the Microsoft HTTP class may be insufficient for file imports. A maximum of 6 MB is recommended for file imports. Files larger than 6 MB can be imported, but performance could be affected. Refer to the Opcenter Execution Medical Device and Diagnostics System Administration Guide or the Opcenter Execution Core System Administration Guide for information on importing files larger than 6 MB.
Note: Review the Private Memory Usage application pool setting for the CamsarAppServer application pool. You may need to increase the value when importing 1000 or more instances (including references). Refer to the Opcenter Execution Medical Device and Diagnostics System Administration Guide or the Opcenter Execution Core System Administration Guide for information on configuring the maximum memory limit for the application pool.

When Importing Modeling Data

The application imports modeling object instances by name. The import process can overwrite existing instances in the application when the import file contains instances with the same name as the existing instances. You can enable an override mode before beginning the import process to prevent instances from being overwritten.

Imports run until the import process is complete. You can modify the Portal settings in Portal Studio to enable a timeout that aborts the import process when a time limit is reached. Refer to the Opcenter Execution Medical Device and Diagnostics Portal Studio User Guide or the Opcenter Execution Core Portal Studio User Guide for information on setting a timeout.

The Portal override mode settings on the Execution Console page allow you to specify whether and how modeling object instances in the application will be overwritten by the modeling object instances in the specified import file. The import process provides three override options for the import:

• Abort - The import process aborts when it detects a modeling object instance in the application with the same name as a modeling object instance in the import file.
• Overwrite - The import process imports the modeling object instances from the import file and overwrites any instances in the application that have the same name.
• Skip - The import process imports the modeling object instances from the input file, but skips importing instances with the same name as an existing instance in the application.
Note: The application uses the Skip override method by default if you do not select an override method.

The application displays all the modeling object instances available for import from the specified XML file in the Object Type Name grid on the Instance Selection page. You can delete the instances you do not want to import. Select the instances in the Object Type Name grid and click the Delete selected row button.

An error will occur when importing modeling object definitions without having first imported the modeling object definitions upon which they depend. For example, an object may depend upon another object if one object defines an instance that populates a required field on the other object's page. You must import the modeling objects that other objects depend upon first, and then import the dependent objects in a separate import.

For example, when importing instances of Approval Template and Approval Decision List, instances of the Approval Decision List should be imported first because the instances of Approval Template expect the Approval Decision List instances to be present in the system during the import process. Instances of Approval Template should then be imported in a separate import process.

Note: The application does not import filter tag associations. For example, if you export a Product modeling object instance that contains a filter tag association, the filter tag association is not included when you import the instance. You can export and import Filter Tag instances.

You can customize the labels that appear on the import pages using Portal Studio. Refer to the Opcenter Execution Medical Device and Diagnostics Portal Studio User Guide or the Opcenter Execution Core  Portal Studio User Guide for information.

Import Failures

Completion messages will indicate failure of any instance imports. Importing an instance will fail if the instance already exists in the application and you prevented overwrites during import.

Pageflow for Manually Importing Modeling Data

The pageflow for importing Modeling data manually consists of 4 pages:

• Data Transfer Type, where you select the manual import
• Instance Selection, where you select the xml file to import and the specific modeling object instances to import
• Execution Console, where you specify the override mode, if any, and start the import
• Execution Summary, where you can view summary and error information for the import

How to Manually Import Modeling Data

Follow these steps to manually import Modeling data:

1. Open the Export/Import page from the Export/Import menu. The Export/Import Data Transfer Type page appears.
2. Complete the required fields on the pageflow using these procedures:
• "How to Select the Data Transfer Type"
• "How to Select the Modeling Object Instances to Import"
• "How to Start the Import"
3. Click Finish to complete the import process. The application closes the Export/Import page and displays the Execution Summary page where you can view a summary of the import and any errors that occurred.
