Defining User Codes

User Codes are lists of codes specific to the requirements of each installation and can provide valuable tracking and reporting information. User-defined codes enable you to specify a list of allowable values for a specific field.

Most user codes consist of three attributes: name, description, and notes. Some user codes include more attributes for additional functionality or validation. Additionally, some user codes can have related WIP messages if they are associated with a field from a container definition.

When Defining User Codes

The name for each user code must be unique within its type. This value is used as an alternate key for look-up, data entry, and validation.

You can also create new user code types as needed.

Refer to "Common Fields on Modeling Pages" for information on the fields common to all modeling objects.

Bill Type is used to distinguish the intent of this bill structure. Several bills can exist for the same product. Bills can be at higher or lower levels of resolution depending on the department that is using the bill. Rollup Reason

Container quantity changes are recorded at the lowest level (child) container and can use more than one reason code. The quantities are then rolled up to intermediate and top-level parent containers. When the rollup occurs, only one reason code is applied to the parent containers. This is the Rollup Reason.

Rollup Reason is an optional field in the Rollup Reason Group modeling definition. Rollup reasons are referenced by the Operation modeling object, but only if you specify a Rollup Reason Group.

Scrap Reason

Reason that may be given for declaring a container as scrap.

Sell Reason

Reason used to record an internal transfer of containers that will not impact yield negatively. Sell Reason is the reason given for such a transfer; for example, a lot can be recorded as sold to Engineering to perform tests on a new process.

Sell Reason is an optional field in the Sell Reason Group modeling definition. Sell reasons are referenced by the Operation modeling object, but only if you specify a Sell Reason Group.

Note: Refer to the Opcenter Execution Core Inline Statistical Process Control Reference Guide for information.

Substitution Reason

Reason that the product actually used differed from the product specified for use.

Training Record Status

Status indicating whether the employee is qualified to perform the task. Refer to "Defining Training Record Statuses" for information.

User Codes with WIP Messages

This table describes the user codes that can have related WIP Messages.

User Code

Description

Hold Reason

Associating a Hold Reason code with a container prevents transactions from being performed for that container. Hold Reason codes are user-defined to provide multiple descriptions (reasons) for Holds.

Hold Reason is a required field in the Hold and Hold (Multiple) transactions. A hold reason is required when recording a production event when the user indicates the affected containers should be held.

Hold Reason is an optional field in the following transactions:

• Container Search
• Release

Owner

Every container has an associated Owner code. The Owner code is used as a grouping to separate containers for inquiry and processing. Examples of Owner code are Manufacturing, Engineering, Prototypes, Sales Samples, and so on.

Owner is a required field in the Start and Start - Two Level transactions.

Owner is an optional field in the Container Maintenance transaction.

Priority Code

An instance of a Priority Code is one method used to assign a processing priority to a container. Each Priority Code includes a description and a Relative Priority value. The Relative Priority value is used to sequence Containers for dispatching (independent of the Code name or description). Priority Codes are only one of the algorithms used to determine a numeric value for Scheduling and Dispatching.

Priority Code is an optional field in the Container definition.

Product Type

Product Type Codes are used to differentiate between types of product such as WIP, Raw Material, Finished Goods, and so on.

Product Type is a required field in the Product definition.

Rework Reason

An instance of a Rework Reason is used to provide a list of reasons for Rework. The code is associated with a container when it enters a rework loop. It is then associated with all throughput transactions generated by the container while it is in the rework loop. This provides, for example, a means of differentiating between first pass and rework throughput in reporting.

Rework Reason is an optional field in the definition of:

• Operation
• Rework Reason Group

Rework Reason is a required field in these transactions:

• Rework
• Transfer Rework

Shipping Reason

Shipping Reasons are associated with container shipment information.

Start Reason

Each container has an associated Start Reason code. Start Reason codes are available for selection criteria on WIP Status Inquiries and for transaction reporting (based on the transaction history).

Start Reason is a required field in the Start and Start - Two Level transactions.

Start Reason is an optional field in the Container Maintenance transaction.

UOM

Abbreviation for unit of measure.

Unit of Measure is associated with container quantity. Refer to "Defining Units of Measure" for information.

UOM is an optional field in the definition of:

• Product
• Product Family
• Mfg Order
• Sales Order

UOM is an optional field in these transactions:

• Container Maintenance
• Resource Thruput
• Start
• Start - Two Level

How to Define a User Code

Follow these steps to define a User Code:

1. Open the User Code Group page. The User Code Group page appears within the Modeling page.
2. Click New. Blank fields appear for you to define a new instance.
3. Enter a name for the user code in the Name field.
4. Enter optional information according to your business requirements. Refer to "Common Fields on Modeling Pages" for information on the optional fields.
5. Click Save. The application saves the modeling object and displays a success message.
