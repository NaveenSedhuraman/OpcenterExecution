Defining Roles

A Role defines a job function with an inherent set of permissions that grants access to all or to parts of the Portal. A Portal administrator assigns roles to employees to grant them access to the parts of the application needed to perform their job. For example, you could define the role of Manufacturing Shift Supervisor and give that role the permission to start a container.

Understanding Roles

A role consists of sets of permissions and permission access modes. A permission is an authorization to perform an action; for example, the permission to start a container or initiate a request. A permission can also be considered a transaction or page.

An access mode specifies the level of authorization for the assigned permission. For example, if a user needs to perform the Start transaction, that user must be associated with a role that includes a permission for the Start transaction and have the access mode of Perform assigned for that permission. The application assigns all access modes by default when you assign a permission.

Roles are only valid for those organizations to which they are assigned within the Manufacturing domain. In the case of an entity with no organizational context, such as transactions, you do not have to specify an organization when assigning a role to a user. Also, roles are valid for sub-organizations only if the administrator propagates the roles to those sub-organizations.

Note: Permissions for extensions to the application (such as new pages) are not added automatically to the default services. You must grant permissions manually for these roles.

Role contains the optional modeling definitions:

• Employee

Role is a required field in these modeling definitions:

• Approval Template, when adding approvers
• ESig Requirement*, when adding electronic signature requirements
• Triage Spec, when adding triage items

Role is an optional field in these modeling definitions.

• Change Mgmt Spec*
• E-mail Distribution
• Employee
• ESig Role Group*
• Organization

*An asterisk denotes the object requires a separately licensed module.

Role Type enables you to specify the type of role you are defining: Change Management, Quality, or Shopfloor. For example, you can set the Role Type for the default role Package Creator to Change Management and the Role Type for Default Quality to Quality.

Note: You can update queries in Designer if you want to filter Role pick lists by Role Type. Contact Product Support via the Global Technical Access Center (GTAC) for information.

When Assigning Permissions

Selecting a permission type from the Permission Type Filter field updates the Assigned Permissions grid and the Available Permissions grid to show only those permissions for the specified permission type. The application maintains all the permissions assigned for the permission types in the current instance, and displays those assigned and available permissions when you view a previously selected permission set. The application saves the permissions to the role definition only when you click Save.

For example, selecting the Container permission type updates the Available Permissions grid to display container-related transactions. If you assign some permissions to the role and then select a different permission type, the application updates the Available Permissions grid to display only permissions related to that permission type. The application clears all container-related permissions from the Assigned Permissions grid, but it does not delete them. The application displays all the container-related assignments when you re-select the Container permission type.

You can select one or more available permissions to assign individually or select all available permissions to assign at once. Selecting the check box in the header row of the Available Permissions grid selects all displayed permissions automatically. The Assigned Permissions grid functions similarly, allowing you to remove assigned permissions individually or all at once.

Note: The Assigned Permissions grid displays all available permissions without paging regardless of global grid settings in Portal Studio. Refer to the Opcenter Execution Core Portal Studio User Guide or the Opcenter Execution Medical Device and Diagnostics Portal Studio User Guide for information.

When Managing Roles

You can manage role memberships from two modeling pages: Role and Employee. If you make changes to the role membership on the Role page, the application displays the updated role membership information for the relevant employee on the Employee page. Likewise, if you update role membership information on the Employee page, the application displays the updated information when you display that role in the Role page.

Roles and Electronic Signatures

Role is a required modeling object when implementing electronic signatures for shop floor and quality record transactions. An electronic signature is the equivalent of a handwritten signature and denotes accountability for the performance of a particular transaction. You can view recorded electronic signatures for a container or quality record in the manufacturing audit trail. Refer to "Electronic Signatures for Modeling Introduction" for information on implementing electronic signatures.

Note: Electronic signatures functionality is a separately licensed module.

How to Define a Role

Follow these steps to define a Role and its permissions:

1. Open the Role page. The Role page appears within the Modeling page.
2. Click New. Blank fields appear for you to define a new instance.
3. Enter a name in the Role field.
4. Select a permission type from the Permission Type Filter field. The Available Permissions grid is updated to show the permissions for the specified permission type.
5. Select the permissions to assign using one of these methods:
• Select the Permission Name check box in the header row of the Available Permissions grid to select all permissions.
• Select the check box for each individual permission.
6. Click Add to assign the selected permissions. The permissions are added to the Assigned Permissions grid and all of the access modes are selected by default.
7. Repeat steps 4-6 to add additional permissions for this role.
8. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
9. Click Save. The application saves the modeling object and displays a success message.

How to Assign a Role to an Employee

Follow these steps to assign a Role to an employee:

1. Perform the “How to Define a Role” procedure.

Or

Select an existing Role instance.

2. Click Add in the Employee Role Memberships grid. A new row appears.
3. Select an Employee.
4. Select the Organization in which the role will be valid for the employee if necessary.
5. Select the Propagate check box if you want the application to make the role valid for all of the specified organization’s sub-organizations.
6. Repeat steps 2-5 to assign this role to another employee.
7. Click Save. The application displays a success message indicating the modeling object was updated.
