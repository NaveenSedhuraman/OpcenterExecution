Defining Change Mgt Settings

Change Mgt Settings enable you to configure various global settings for change management. These settings include:

• Maximum Where Used Instances to be passed - The maximum number of modeling object instances the application can pass to the Where Used function when the package owner or collaborator assigns content to a package.
• Package Creation Template - Template used to create a package for an item and revision sent from Teamcenter to Opcenter EX MDD or Opcenter EX CR.
• Notification Events - Notifications sent to employees when certain events occur for a change management package. These events include:
• Change Mgt Approval Assignment - Notifies employees required to approve a change management package when the package is routed for approval.
• Change Mgt Approval Delegation - Notifies assigned and target employees that a package approval task has been reassigned.
• Change Mgt Collaboration Assignment - Notifies collaborators that they can assign content to a change management package.
• Change Mgt Collaboration Delegation - Notifies assigned and target employees that a package collaboration task has been reassigned.
• Change Mgt Owner Collaboration Assignment Complete - Notifies package owners when collaborators finish assigning content to a package.
• Package Approval Reminders - Reminder notifications sent to the owner, approvers, or approver escalation recipients when a package approver has upcoming, due, or overdue approvals.
• Content Collaboration Reminders - Reminder notifications sent to the owner, collaborators, or collaborator escalation recipients when a content collaborator has upcoming, due, or overdue collaboration work.
Note: Refer to the Opcenter Execution Medical Device and Diagnostics Change Management User Guide or the Opcenter Execution Core Change Management User Guide for information on change management e-mail notifications.

When Defining Change Mgt Settings

The maximum Where Used Instances default is 1. The application displays a warning message if you enter a value less than 1.

Note: Increasing the maximum Where Used Instances impacts performance. The recommended number is 5. The application accepts values greater than 5 but this may affect performance.

Opcenter EX MDD or Opcenter EX CR uses the Package Creation Template when it creates a package for an item and revision sent from Teamcenter to Opcenter EX MDD or Opcenter EX CR. The package includes the product and revision that contains Bill of Process (BOP) and Bill of Material (BOM) information. Specifying a template populates the Create Package page with the predefined package details, change management workflow, and deployment details if you set the Change Mgt Settings field on the factory associated with the employee starting the package. Refer to "Defining Package Creation Templates" and "Defining a Factory" for information.

Note: This functionality is available only when integrating with Teamcenter. Refer to the Opcenter Execution Core Teamcenter Integration Guide for information.

The application does not require you to add an e-mail notification when defining change management settings. However, you must configure e-mail notifications if you want the application to send notifications to employees when events occur for a package. The application requires you to select both a Notification Type and an E-mail Message when you add an e-mail notification. You define e-mail messages using the E-mail Message modeling object.

The notification type for change management can only be selected from the Notification Type picklist on the Change Mgt Settings page. You cannot select these notification types from the Notification Type list on the Organization page.

A system administrator can reassign a task from one employee to another. For example, the administrator can delegate package approval or collaboration work to another employee if the employee to which the work was originally assigned will be on vacation when the work is due. Select the Change Mgt Approval Delegation or Change Mgt Collaboration Delegation notification types and select the corresponding e-mail messages to be sent to the assigned and target employees when the work is reassigned. Refer to the Opcenter Execution Medical Device and Diagnostics Shop Floor User Guide or the Opcenter Execution Core Shop Floor User Guide for information on delegating tasks.

Change Mgt Settings is an optional field you can set when defining a Factory modeling object instance. Factory is an optional field when defining Employee instances. Therefore, you can link an employee to change management settings by setting the Factory field on the Employee page to the factory associated with the change management settings.

You must define the SMTP transport to be used for e-mail notifications. The Factory modeling object instance must indicate the SMTP transport defined for e-mail notifications. Ensure that the Employee modeling object instances indicate the factory where the SMTP transport is associated. Refer to the Opcenter Execution Medical Device and Diagnostics Change Management User Guide or the Opcenter Execution Core Change Management User Guide for information on change management e-mail notifications.

When Defining Reminder Notifications

You can set up reminder notifications if you want to send e-mail messages to certain employees when a package approver or content collaborator has upcoming, due, or overdue work. The application sends reminder notifications for approval and collaboration tasks based on their due date.

Due date is determined by assignment date, Duration, and Period as specified by the package owner when the work is assigned. The approvers and collaborators are expected to complete the work within the specific time frame. The due date is determined by Duration and Period if the package owner sets these fields when assigning the work. The due date is the assignment date and time if the package owner does not set Duration and Period. A task is due the moment it is assigned if Duration and Period are not set.

You must specify the e-mail message, recipient, and days (in relation to the due date) that you want the application to send the messages.

You define e-mail messages using the E-mail Message modeling object. You can use the Text Variable modeling object in the e-mail message if you want to set up dynamic messages. For example, you can define an e-mail message for the package owner with the body text "Package <package name> has incomplete approvals." Refer to "Defining E-mail Messages" and "Defining Text Variables" for information.

You can specify that these employees receive the approval reminder notifications:

• Owner - Notifies the owner of the package.
• Approvers - Notifies each approver who has an incomplete task.
• Approver Escalation Recipients - Notifies the escalation recipients for any approver who has an incomplete task.

You can specify that these employees receive the content collaboration reminder notifications:

• Owner - Notifies the owner of the package.
• Collaborators - Notifies each collaborator who has an incomplete task.
• Collaborator Escalation Recipients - Notifies the escalation recipients for any collaborator who has an incomplete task.
Note: Ensure that the E-mail Address field is complete in each recipient's Employee modeling object instance or the recipients will not receive the e-mail notifications.
Note: You define escalation recipients using the Employee modeling object. For example, you can send an e-mail notification to a content collaborator's manager if the collaborator's work is overdue. The application will not send e-mail notifications to escalation recipients if you do not select them in the Escalation Recipients grid in the approver's or collaborator's Employee modeling object instance.

You must specify the number of days before or after the due date to send the notifications, or you can send the notification on the due date. For example, enter 3 for Days Before Due to send to the notification three days before the approval or collaboration work is due. Enter 1 for Days After Due if you want to send the notification one day after the work is due. Enter a 0 for either Days Before Due or Days After Due to send the notification on the due date.

You cannot specify both Days Before Due and Days After Due for the same row in the Package Approval Reminders or Content Collaboration Reminders grids. If you enter a value in the Days After Due field and there is already a value in the Days Before Due field in the same row, the application clears the value from the Days Before field. Similarly, if you enter a value in the Days Before Due field and there is already a value in the Days After Due field in the same row, the application clears the value from the Days After field.

The application sends reminder notifications daily until the work is complete if you select the Repeat Daily check box. The following occurs depending on the type of recipients you specify:

• The package owner receives reminder notifications until all approvals or collaboration work is complete.
• The approver or collaborator receives reminder notifications until he or she completes the approval or collaboration work.
• The escalation recipients for the approver or collaborator receive reminder notifications until the approver or collaborator completes the approval or collaboration work.

The application sends reminder notifications according to a scheduled business rule. Siemens provides a default scheduled business rule named SBR_NOTIFICATIONS. This rule runs every 24 hours starting at 4:00 a.m. The reminders are sent each time the scheduled business rule runs and the reminder is eligible. For example, a system administrator may set a reminder notification to be sent to a collaborator one day before the collaboration work is due at 9:30 a.m. on Friday. The application runs the default scheduled business rule at 4:00 a.m. on Thursday and sends the reminder notification to the collaborator.

A system administrator can also define additional scheduled business rules. For example, an administrator may define a scheduled business rule to send these e-mail messages every six hours starting at 12:00 a.m. The application runs the scheduled business rule at 12:00 a.m., 6:00 a.m., 12:00 p.m., and 6:00 p.m. on Thursday and sends the reminder notification to the collaborator each time. Refer to "Defining Scheduled Business Rules" for information.

How to Define Change Mgt Settings

Follow these steps to define Change Mgt Settings:

1. Open the Change Mgt Settings page. The Change Mgt Settings page appears within the Modeling page.
2. Click New. Blank fields appear for you to define a new instance.
3. Enter a name for the change management settings in the Change Mgt Settings field.
4. Enter a maximum number of where used instances in the Maximum Where Used Instances field.
Note: The application default is the minimum number of Where Used Instances allowed. The recommended number of Maximum Where Used Instances is 5. However, the application accepts values greater than 5, but this may affect performance.
5. Perform the "How to Add Notification Events" procedure if you want to add notification events.
6. Perform the "How to Add Package Approval Reminders" procedure if you want to add package approval reminders.
7. Perform the "How to Add Content Collaboration Reminders" procedure if you want to add content collaboration reminders.
8. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
9. Click Save. The application saves the modeling object and displays a success message.

How to Add Notification Events

Follow these steps to add notification events:

1. Perform the "How to Define Change Mgt Settings" procedure.

Or

Select an existing Change Mgt Settings instance.

2. Expand the E-mail Notifications section if necessary.
3. Click Add new row in the Notification Events grid. A new row appears.
4. Select a Notification Type.
5. Select an E-mail Message.
6. Repeat steps 2-5 to add additional notification types.
7. Click Save. The application saves the modeling object and displays a success message.

How to Add Package Approval Reminders

Follow these steps to add package approval reminders:

1. Perform the "How to Define Change Mgt Settings" procedure.

Or

Select an existing Change Mgt Settings instance.

2. Expand the E-mail Notifications section if necessary.
3. Click Add new row in the Add Package Approval Reminders grid. A new row appears.
4. Select an E-mail Message.
5. Select a Recipient.
6. Enter an integer value greater than or equal to 0 in the Days Before Due field.

Or

Enter an integer value greater than or equal to 0 in the Days After Due field.

7. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
8. Repeat steps 3-7 to add additional package approval reminders.
9. Click Save. The application saves the modeling object and displays a success message.
