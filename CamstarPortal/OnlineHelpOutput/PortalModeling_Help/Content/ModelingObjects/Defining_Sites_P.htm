Defining Sites

A Site is an independent Opcenter EX MDD or Opcenter EX CR entity within a manufacturing organization. While a factory is a physical facilities concept, a site is the name of an installation of Opcenter EX MDD or Opcenter EX CR containing an Application Server. The name is used by other applications that send transactions to and from this site. A factory can have one or more sites, and a site can run one or more factories.

When Defining a Site

A Site definition contains the optional data transport modeling definition. SMTP data transport definitions are available for selection from the Default Data Transport and E-mail Transport fields.

Site is an optional field in the following modeling definitions:

• Enterprise, you can specify a site in the XML Connect Proxy Site field
• Shipment Destination, you can specify a site in the Remote Site field

Additionally, Site is:

• A required definition when configuring Opcenter EX MDD or Opcenter EX CR for remote e-mail.

When Associating an Employee with a Site

Associating an employee (user) with the site definition enables that employee to perform certain remote transactions. For example, the employee you associate with this site can perform the Remote Start transaction to create a container on the site remotely. The employee must be assigned a role that has permissions to perform transactions remotely.

This table describes the remote transactions the associated employee can perform.

Transaction

Description

RemoteStart

Creates a container or multi-level container structure on the remote site.

RemoteReceive

Subsequent (subordinate) service triggered by a Ship transaction on a remote site. The purpose of RemoteReceive is either to create a new container (through RemoteStart), or to reactivate an existing container (through Ship) on the destination site for further processing. If the same container does not exist on the destination site, RemoteReceive will create a new container. If the destination site already has the same container the shipping system is sending over, then the RemoteReceive will reactivate the container on the destination site and update it with new information.

Note: RemoteReceive is a compound transaction that may consist of RemoteStart, or Ship and any other service transactions desired.

Ship

Sends a container from one factory (site) to another or from a factory (site) to a customer. The ship transaction contains all of the validations and "To Step" logic found in MoveNonStd.The container can be sent to any workflow and any step in the other factory. A ship transaction can trigger a RemoteReceive compound transaction also, which will be sent over to the destination factory for further processing.

InventoryTransfer

Sends WIP information from one factory (site) to another or from a factory (site) to a customer. InventoryTransfer can be used as an independent, standalone service to send WIP information across factories.

The InventoryTransfer transaction contains all of the behaviors and validations in the Ship transaction except the following:

• InventoryTransfer allows a closed container to be reopened when the container is shipped back from the destination site to the original sending site.
• No Move-In would be required on the local sending site when the shipped away container is returned.
• No production credit is transferred when the shipped away container is returned.
• InventoryTransfer is used in the RemoteReceive Compound transaction currently as the means to return a container to a site where the same container has existed, or still exists.

How to Define a Site

Follow these steps to define a Site:

1. Open the Site page. The Site page appears within the Modeling page.
2. Click New. Blank fields appear for you to define a new instance.
3. Enter a name for the site in the Site field.
4. Select the employee's user name in the User field.
5. Enter the employee's Windows logon password in the Password field. The password is encrypted as you type.
6. Enter the site's URL in the URL field.
7. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
8. Click Save. The application saves the modeling object and displays a success message.
