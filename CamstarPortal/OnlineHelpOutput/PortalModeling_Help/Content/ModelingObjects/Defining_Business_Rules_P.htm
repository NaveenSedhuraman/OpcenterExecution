Defining Business Rules

The Business Rule modeling object is used to define a condition to be met and one or more actions to be executed if that condition is true. You can associate business rules at several points during the quality management process to control when certain actions occur automatically. For example, you can create a business rule for changing owners during triage.

Opcenter EX MDD and Opcenter EX CR provide these default business rules:

• BR_ACTIVATION - This rule links to the default business rule handler BRH_ACTIVATION in the Handler grid.
• BR_DEPLOY - This rule links to the default business rule handler BRH_DEPLOY in the Handler grid.
• BR_DEPLOYSTATUS - This rule links to the default business rule handler BRH_DEPLOYSTATUS in the Handler grid.
• BR_NOTIFICATIONS - This rule links to the default business rule handler BRH_NOTIFICATIONS in the Handler grid.
Note: Refer to "Defining Business Rule Handlers" for information.

When Defining Business Rules

Business Rule contains the optional modeling definition, Business Rule Handler.

Business Rule is an optional field in the Triage Spec modeling definition. The Usage field must be set to QualityObject before it appears as an option in the Triage Spec modeling definition.

Business Rule is an optional field in the Spec modeling definition. Only business rules that have a Usage type of Container are available in the Business Rule Txn Map grid on the spec. Refer to "Adding Business Rules to a Spec" for information.

The application enforces business rule usage at the point when business rules are assigned to other objects.

Configure the conditional statement using the Unified Expression syntax. This syntax allows you to create simple or complex expressions that can be evaluated based on where the business rule is assigned.

When you add actions to the business rule, you can set the parameters for individual actions to expressions that the application will evaluate when executing the business rule. This allows you to overwrite values in the action script or service.

Schedule business rules so that they are evaluated at pre-determined times in the future. You can schedule the rules to be evaluated once or on a recurring basis. For example, you can send a warning to a disposition approver if the disposition is not approved at a specific time.

When Viewing Existing Business Rules

Siemens recommends that you open and save all existing business rules that reference handlers containing parameters that were added, deleted, or updated. Otherwise, errors may occur when the associated business rules are executed.

The application synchronizes parameters defined for the business rule with those defined for associated business rule handlers when you open an existing business rule. If there are no differences between the parameters, no changes are necessary. If an associated handler has parameters that are not included in the business rule, the application adds the parameter to the business rule automatically. If you have removed parameters from an associated handler, the parameters are not displayed in the business rule.

The application displays instance IDs manually before you can save the business rule.

Usage

Field specifying where the business rule can be assigned and used. The application uses the usage type to determine how to evaluate the business rule parameters at run time. Options include the following:

• Change Package
• Container
• Employee
• Event
• Factory
• QualityObject

Scope

Type of processing based on the executing user’s role and security rights. The default is Global.

Selecting Global indicates that the application process the business rule regardless of the executing user’s role and security rights.

Selecting Local indicates that the application process the business rule only if the executing user’s organization, role, and security rights allow.

The application enforces the selected scope with the exception of a business rule schedule. Because user roles and security rights are not an issue when the application executes a business rule schedule, the application executes the business rules assuming a Global scope.

Required

Always Execute

Check box indicating that the actions in the Handler grid be executed every time.

Although this is an optional field, the application requires that you either enter a condition in the Condition field or select this check box before it will save the business rule. If you select this check box and enter a conditional expression, the application ignores the condition and always executes the actions.

Optional

Condition

Condition that the application will evaluate to determine whether to execute the action in the Handler field.

Although this is an optional field, the application requires that you either enter a condition or select the Always Execute check box before it will save the business rule. If you enter a conditional expression and select the Always Execute check box, the application ignores the condition and always executes the actions.

How to Define a Business Rule

Follow these steps to define a Business Rule:

1. Open the Business Rule page. The Business Rule page appears within the Modeling page.
2. Click New. Blank fields appear for you to define a new instance.
3. Enter a name for this business rule in the Business Rule field.
4. Select the object to which the business rule can be assigned from the Usage field.
5. Select the processing scope from the Scope field.
6. Do one of the following:
• Select the Always Execute check box to indicate that the application perform the actions in the Handler field every time.
• Enter the condition to be evaluated in the Condition field using Unified Expression syntax.
7. Click Add new row in the Handler grid. A new row appears.
8. Select a handler to be used as the action. The application displays the handler in the grid. Additionally, if there are any parameters defined for the handler, those appear in the Parameters grid.
9. Repeat steps 7-8 to add additional handlers.
10. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
11. Click Save. The application saves the modeling object and displays a success message.

How to Override Handler Parameters with Custom Values

Follow these steps to override handler parameters with custom values:

1. Perform the "How to Define a Business Rule" procedure.

Or

Select an existing Business Rule instance.

2. Locate the handler parameter that you want to customize in the Parameters grid and then select it.
3. Enter an expression in the Expression field. The application will evaluate the expression at run time to set the value for that parameter.
4. Repeat steps 2-3 to modify additional parameters.
5. Click Save. The application displays a success message indicating the modeling object was updated.
