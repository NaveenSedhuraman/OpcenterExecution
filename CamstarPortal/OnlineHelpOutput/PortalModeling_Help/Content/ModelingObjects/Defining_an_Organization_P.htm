Defining an Organization

An Organization is a business entity that is separated for processing and reporting purposes. An organization is not associated with an enterprise or a factory.

When Defining an Organization

Organization contains these optional modeling definitions:

• Checklist Template
• Classification
• E-mail Distribution
• E-mail Message
• Failure Mode Group
• Numbering Rule
• Owner
• Organization
• Role
• SMTP Transport
• Subclassification
• Triage Spec
• UI Preference

If you specify these optional modeling definitions, certain fields are required to define them.

Organization is an optional field you can specify when defining Employee and Role.

While the numbering rule fields are not required entries when creating or updating an organization, you must create a record of the corresponding category for that organization before you can create an event or a nonconformance.

A Primary Organization must be assigned to the Employee before you can record a production event. In addition, the primary organization must have a Role and Owner for the logged-in user assigned to the appropriate Classification and Subclassification in the Event Classification Spec Maps grid.

Security Processing Criteria

Organizations are required for Opcenter EX MDD and Opcenter EX CR security administration. At least one organization is required to set up security administration. Opcenter EX MDD and Opcenter EX CR are installed with a default organization of Corporate.

An employee is assigned permissions through his or her association with a role (job function) that is valid within the context of the organization, and these associations determine the employee’s system privileges.

A user must have a role in the organization, either directly or indirectly through role propagation, to perform any action.

Refer to the Opcenter Execution Medical Device and Diagnostics System Administration Guide or the Opcenter Execution Core System Administration Guide for information on default employee logins, which are loaded when Opcenter EX MDD or Opcenter EX CR is installed.

Quality Record Processing Criteria

Defining organizations for quality record processing enables you to specify an organization’s processing criteria, including:

• Classification/Subclassification mappings for creating quality records
• UI preferences for displaying quality record information in the Manufacturing Audit Trail
• Category-specific quality record processing criteria, such as triage spec, category, role, and owner
• Electronic signature requirements for quality transactions
• Numbering rule maps for quality records
• SMTP Transport definitions
• E-mail notification information
• Transactions not allowed on containers associated with generic or production events
• Transactions, that when executed, add affected containers to quality records

When you create a quality record (event), Opcenter EX MDD or Opcenter EX CR uses the record’s organization and category (type) to determine the criteria for processing that record. For example, if the quality record is an event belonging to the Corporate organization, the application looks at the organization definition to determine the triage spec to use to process a quality record and the numbering scheme to use when generating a unique ID for a quality record.

Notifications are a way of identifying e-mails to be sent when specific events/actions occur in the application. They can be used to alert a recipient of a required action or to provide an important status update. For example, you can specify an e-mail to be sent to the new owner when ownership changes for an event.

You can also use the Organization object to specify the quality transactions that require an electronic signature.

You can specify the transactions you want to prohibit for containers associated with generic or production events. You can also specify that the application add affected containers to a quality event automatically when a transaction is executed on a related container with affected material.

The next several topics provide more detailed information on how the application uses the Organization object.

Role and Owner Assignment

Quality record processing requires that a role and owner be assigned to an event before a user can submit the event. If the event does not have a role and owner assigned at the time of submittal, an error message is displayed that indicates the owner and role are not assigned and prevents the user from submitting the event.

The application provides multiple opportunities for role and owner assignment:

• Automated business rules (applied automatically during triage if associated with the triage spec used to process the event)
• Event Classification Spec Map (defined as part of the Organization modeling object)
• Category Processing Map (defined as part of the Organization modeling object)

The first opportunity for role and owner assignment occurs during the automatic execution of business rules through triage. If one or both values are missing, the application checks the Event Classification Spec Map first. (It has priority over the Category Processing Map.) If role and owner are specified here, the application uses these values. If not, the application uses the role and owner values specified for the Category Processing Map.

Note: If during automatic business rule execution, only one of the values is set, the application will overwrite that value with the value specified in the Event Classification Spec Map or the Category Processing Map as applicable. For example, if the role value is set during triage and the owner value is empty, the application checks the role and owner values specified in the Event Classification Spec Map. If the Event Classification Spec Map contains role and owner values, the application will overwrite the role value set during triage with the role value specified in the Event Classification Spec Map.

The Event Classification Spec Map is required for events and nonconformances.

Portal pages and pageflows are defined in the Portal Studio.

You can create classifications and subclassifications for your organization using the Classification and Subclassification modeling objects.

When Defining Category Processing Maps

Quality record processing uses category processing maps to assign the following to quality records in a certain category:

• Triage specs
• Role and owner (if not assigned previously)

When defining a category processing map, the category itself is the primary key and must be unique. For example, you can create only one category processing map with a category of event.

When Configuring Electronic Signature Requirements

Typically, when specifying quality transactions for which you want to collect electronic signatures, you need to add only that quality service to the Electronic Signature Txn Map on the organization modeling object. For example, if you want to collect an electronic signature when a user cancels an approval for a quality record resolution, you would add the Cancel Approval Sheet quality service to the Electronic Signature Txn Map.

The exception is event creation. If you want to require electronic signatures for event recording pageflows, then you must specify not only the CreateEvent service in the Electronic Signature Txn Map, but you must also add the UpdateEvent service.

If you do not add the update services to the Electronic Signature Txn Map, then the user will be able to bypass the electronic signature requirement when executing the event recording pageflows.

ESig Requirement for Assign Approval Sheet

When an electronic signature is configured for Assign Approval Sheet, it is only related to the assignment/routing of resolution approvals for quality records.

When Defining Transaction Defaults

You can specify the transactions you want to prohibit for containers associated with generic or production events. Clicking Add new row in the Production Event Disallowed Transactions grid or the Generic Event Disallowed Transactions grid displays a list allowing you to select a container transaction. The list includes every container transaction, such as Change Qty, Close, Hold, and Move.

The application can add affected containers to a quality event automatically.

You can specify container transactions for which you want the application to add containers to a quality event automatically. The application adds affected containers to a quality event automatically when any of the transactions specified in the grid are executed on a related container with affected material.

These default transactions are available in the Add Affected Containers to Events grid:

• Associate
• Combine
• Component Issue
• Component Remove
• Split

The application adds affected containers to a quality event automatically when any of the transactions you select in the grid are executed on a related container with affected material.

Note: The application adds the affected containers to a quality event when you execute the Combine Container or Combine Qty transactions if you select Combine. Similarly, Split applies to the Split Container and Split Qty transactions.

Refer to the Opcenter Execution Medical Device and Diagnostics Event Management User Guide or the Opcenter Execution Core Event Management User Guide as well as the Opcenter Execution Medical Device and Diagnostics Nonconformance Management User Guide or the Opcenter Execution Core Nonconformance Management User Guide for information on adding affected containers to quality events automatically.

How to Define an Organization

Follow these steps to define an Organization:

1. Open the Organization page. The Organization page appears within the Modeling page.
2. Click New. Blank fields appear for you to define a new instance.
3. Enter a name for this organization in the Organization field.
4. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
5. Click Save. The application saves the modeling object and displays a success message.

How to Add an Event Classification Spec to an Organization

Follow these steps to add an event classification spec to an Organization:

1. Perform the "How to Define an Organization" procedure.

Or

Select an existing Organization instance.

2. Locate the Event Classification Spec Map grid.
3. Click Add new row. A new row appears.
4. Select the appropriate Classification for the event.
5. Select the Subclassification to be associated with the classification.
6. Select the Failure Mode Group to be associated with the event classification and subclassification.
7. Select the default Role to be associated with quality records created with this event classification and subclassification.
8. Select the default Owner associated with quality records created with this event classification and subclassification. You cannot select an owner unless you first select a role.
9. Optionally select a Checklist Template.
10. Click Save. The application displays a success message indicating the modeling object was updated.
11. Repeat steps 3-10 to add another event classification spec to this organization.

How to Remove an Event Classification Spec from an Organization

Follow these steps to remove an event classification spec from an Organization:

1. Perform the "How to Define an Organization" procedure.

Or

Select an existing Organization instance.

2. Locate the Event Classification Spec Map grid.
3. Select a row. The Delete selected row button is enabled.
4. Click Delete selected row. The row is removed from the grid.
5. Repeat steps 3-4 to remove additional event classification specs, if needed.
6. Click Save. The application displays a success message indicating the modeling object was updated.

How to Add Triage Specs to an Organization

Follow these steps to add triage specs to an Organization:

1. Perform the "How to Define an Organization" procedure.

Or

Select an existing Organization instance.

2. Locate the Category Processing Map grid.
3. Click Add. The Add pop-up appears.
4. Select the category from which you want to escalate the item from the Category list.
5. Complete this step if you selected Event from the Category list:
a. Select a Role from the list.
b. Select an Owner from the list.
6. Select a triage spec from the Triage Spec list.
7. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
8. Click OK. The application closes the Add pop-up displays the category and its associated triage spec in the Category Processing Map grid.
9. Repeat steps 3-8 to add additional triage specs.
10. Click Save. The application displays a success message indicating the modeling object was updated.

How to Remove Triage Specs from an Organization

Follow these steps to remove triage specs from an Organization:

1. Perform the "How to Define an Organization" procedure.

Or

Select an existing Organization instance.

2. Locate the Category Processing Map grid.
3. Select a row that contains a triage spec. The Delete selected row button is enabled.
4. Click Delete selected row. The selected row is removed from the grid.
5. Repeat steps 3-4 to remove additional triage specs.
6. Click Save. The application displays a success message indicating the modeling object was updated.

How to Assign an Approval Template to a Quality Record

Follow these steps to assign an approval template to a quality record:

1. Perform the "How to Define an Organization" procedure.

Or

Select an existing Organization instance.

2. Assign an approval template to an existing category or a new category as follows.

If you want to add an approval template to . . .

Then . . .

An existing category
a. Select a row, and then click Edit selected row. The Edit pop-up appears.
b. Go to step 3.

A new category
a. Click Add new row in the Category Processing Map grid. The Add pop-up appears.
b. Select a category from the Category list.

How to Assign a Numbering Rule to a Category for an Organization

Follow these steps to assign a numbering rule to a category for an Organization. Note that you can assign only one numbering rule to each category type:

1. Perform the "How to Define an Organization" procedure.

Or

Select an existing Organization instance.

2. Click Add new row in the Numbering Rule Map grid. A new row appears.
3. Select a category from the Category list.
4. Select a numbering rule for the category from the Numbering Rule list.
5. Repeat steps 2-4 to add numbering rules to additional categories.
6. Click Save. The application saves the numbering rule assignments and displays a message indicating that your updates were successful.

How to Configure Notifications

Follow these steps to configure notifications:

1. Perform the "How to Define an Organization" procedure.

Or

Select an existing Organization instance.

2. Locate the Notification Events grid and click Add new row. A new row appears.
3. Select a notification type from the Notification Type list.
4. Select an e-mail message from the E-mail Message list.
5. Select Append Concierge Msg if you want to append the concierge message to the e-mail message.
6. Either select an e-mail distribution from the E-mail Distribution list or select the Assignee Only check box.
Note: You cannot select both an E-mail Distribution and the Assignee Only check box, but you must select one of them.
7. Click Save. The application displays a success message indicating the modeling object was updated.

How to Configure Electronic Signature Requirements

Follow these steps to configure electronic signature requirements:

1. Perform the "How to Define an Organization" procedure.

Or

Select an existing Organization instance.

2. Scroll down to the Electronic Signature Txn Map grid.
3. Select the Apply Electronic Signature to All Transactions check box if you want to require an electronic signature for all quality transactions.

Or

Specify selected Quality transactions that require an electronic signature as follows:

a. Click Add new row. A new row appears.
b. Click the new row and select a transaction from the Quality Service list. The transaction is added to the list.
4. Select the Require Entry of User ID check box if the employee’s password and user name are required for the electronic signature verification. If only the password is required, leave this box blank.
5. Click Save. The application displays a success message indicating the modeling object was updated.
