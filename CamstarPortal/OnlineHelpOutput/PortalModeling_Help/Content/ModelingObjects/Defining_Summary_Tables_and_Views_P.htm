Defining Summary Tables and Views

The Summary Table Definition modeling object is used to configure views and summary tables. Views and summary tables organize data in a logical, summary fashion. Siemens employs these definitions for Business Intelligence (BI) reporting.

Summary Tables

A summary table is a collection of adjacent fields of data that are permanently stored and read at runtime. It can be left unchanged, be appended to, or be refreshed entirely.

Summary tables can be executed manually or configured to run automatically. If you specify that a summary table should be executed manually, the application disables any recurrence pattern settings previously defined for the summary table.

Views

A view is a way of portraying information in a database. Often thought of as a virtual table, a view does not actually store information itself, but instead pulls information out of one or more existing tables.

Although impermanent, a view may be accessed repeatedly by storing its criteria in a query.

Views must be executed manually.

When Configuring Summary Tables and Views

When configuring a summary table definition, remember these principles:

• Only a database administrator can remove a view or summary table from the database.
• Although the definition uses the DataStore by default, it can be reconfigured to point to the transaction database or to both at the same time. Pointing to the transaction database, or both databases, however, can substantially lower system performance.
• The default recurrence pattern for most summary tables is to run once every 24 hours, starting at midnight (0). The default recurrence pattern for the csiProductionDashboard and csiYield summary tables is to run every four hours, starting at midnight (0).
• The selections for recurrent scheduling are additive. Therefore, you must be careful when making your selections. For example, if you initially select Monday and Saturday, the summary table will only run on Mondays and Saturdays. If you then decide to add day 1 of the month as a Specific Day of the Month, then the summary table will only run on Mondays and Saturdays that are also the first day of the month.
• When a SQL query is tested, only 100 rows are returned for review.
• View or table names entered in the View or Table Name field must be unique.
• You can use the Modeling Audit Trail to view a history of this modeling object.

When Creating a Query

When creating a query that uses the LastRunDate parameter, you must indicate if it is being run against a SQL database or an Oracle database.

Database

Value

SQL

@LastRunDate

Oracle

:LastRunDate

Example

This example shows a query that is being run against a SQL database.

SELECT productid,count(*) CNT

FROM container

WHERE productid IS NOT NULL

AND (originalstartdate >= @LastRunDate OR @LastRunDate IS NULL)

GROUP BY productid

Important: Any time the SQL Statement is modified, you must select the Force Refresh check box and click the Save and Execute Manually button to commit the change. This applies to both tables and views.

When Executing Summary Tables

When executing summary tables, remember these principles:

• You must select the Execute Manually check box if you decide to execute a previously scheduled summary table manually. This causes the Scheduling section to disappear. You must then click the Save and Execute Manually button to execute the table.
• Clear the Execute Manually check box if you decide to re-schedule the table. This causes the Scheduling section to reappear with the original settings.
• Clicking the Save and Execute Manually button does not verify your query like the Save and Test Query button.

How to Define a Summary Table

Follow these steps to define a Summary Table:

1. Open the Summary Table Def page. The Summary Table Def page appears within the Modeling page.
2. Click New. Blank fields appear for you to define a new instance.
3. Enter a name in the Summary Table Def field.
4. Enter query text in the Query Text field.
5. Expand the Scheduling section.
6. Select Summary Table. The page displays check boxes and recurrence pattern fields specific to summary tables.
7. Enter a name for the summary table in the View or Table Name field.
8. Do you want to execute the summary table query manually or configure it to run automatically?

If you want to . . .

Then . . .

Execute the summary table query manually
a. Select the Execute Manually check box. The recurrence pattern fields are hidden.
b. Go to step 9.

Configure the summary table query to run automatically
a. Select the Enable Recurrence check box.
b. Use the Recurrence Pattern and Range of Recurrence options to configure the run times.
c. Go to step 9.
9. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
10. Click Save. The application saves the modeling object and displays a success message.

How to Define a View

Follow these steps to define a View:

1. Open the Summary Table Def page. The Summary Table Def page appears within the Modeling page.
2. Click New. Blank fields appear for you to define a new instance.
3. Enter a name for the view in the Summary Table Def field.
4. Enter query text in the Query Text field.
5. Expand the Scheduling section.
6. Select View. The recurrence pattern information is hidden.
7. Enter a name in the View or Table Name field.
8. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
9. Click Save. The application saves the modeling object and displays a success message.

How to Test the Query

Follow these steps to test a Summary Table Definition Query:

1. Do one of the following:
• Perform the "How to Define a Summary Table" procedure.
• Perform the "How to Define a View" procedure.
• Select an existing Summary Table Def instance.
2. Click Save and Test Query. The UserQueryMaintTest pop-up appears and displays a status message or results of the query.
3. Click Close to close the UserQueryMaintTest pop-up.
4. Modify and test the query again until you achieve the desired result if necessary.
Important: If a summary table query takes a long time to run, it may encounter table locking errors in the SQL Server due to incoming transactions. You can modify the Summary Table SQL to allow for queries that do not lock if this occurs.

How to Execute a Summary Table or View Manually

Follow these steps to execute a Summary Table or View manually:

1. Do one of the following:
• Perform the "How to Define a Summary Table" procedure.
• Perform the "How to Define a View" procedure.
• Select an existing Summary Table Def instance.
2. Expand the Scheduling section.
3. Click Save and Execute Manually. The application saves the modeling object and displays a success message.
