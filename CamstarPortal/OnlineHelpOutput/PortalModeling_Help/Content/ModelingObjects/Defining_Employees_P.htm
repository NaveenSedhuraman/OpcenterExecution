Defining Employees

An Employee is a user who performs activities in the system such as maintaining modeling definitions, making inquiries, managing security administration, and/or performing shop floor transactions. A system administrator uses security administration features to grant privileges to employees so they can access those activities.

Employees must have an Active Directory account and the employee names must match Active Directory login IDs. Contact your Active Directory system administrator for the exact syntax of user names.

Employees can be defined without accounts, but those employees cannot log in. They can be entered as the employee who performed a container or resource transaction. For example, a supervisor with an Active Directory account (and the appropriate permission) can log in to the application and access the Start page. The employee name recorded on the Start page can be the supervisor or another employee, whoever actually performed the work of starting the container. This employee may or may not have an Active Directory account.

Session values can be set for each employee to limit the view of someone who is performing transactions. The employee sees only what he or she is supposed to see, and has a shorter list of values to search from. This limits searching time and presents fewer opportunities for mistakes.

Security Model

The security model for Opcenter EX MDD and Opcenter EX CR is based on a role-based access control model. The system administrator assigns role memberships to an employee to control the employee’s access to all or parts of the system. When you specify a primary organization and role memberships for an employee, you are setting up the security rights for that employee. A user must have a role in the organization, either directly or indirectly through role propagation, to perform any action.

Roles are defined on the Role page. You can assign role memberships on the Employee page or on the Role page. However, you can only assign permissions for that role on the Role page.

The security guide for your product includes a chapter on security administration. It discusses permission types and modes, how the authentication and authorization process works, and how user permissions are validated. Refer to the Opcenter Execution Medical Device and Diagnostics Security Guide or the Opcenter Execution Core Security Guide depending upon your product.

Default Employees

Opcenter EX MDD and Opcenter EX CR provides three employee definitions by default:

• Administrator
• CamstarAdmin
• InSiteAdmin

Use one of these employee names when initially logging on to the application.

The default Portal menus are assigned to the Administrator, CamstarAdmin, and InSiteAdmin employees automatically in an initial implementation of Opcenter EX MDD or Opcenter EX CR, or when converting or updating to a new version and upgrading default instance data. Use the Portal Menu Definition field on the Employee page to assign Portal menus to other employees.

Default Roles

Siemens also provides several predefined roles for the default Corporate organization. The default employees are assigned membership in all or some of the default roles. Additional default roles are provided to users who have licensed Change Management.

Refer to the Opcenter Execution Medical Device and Diagnostics Security Guide or the Opcenter Execution Core Security Guide for information on how employees and roles are used in security.

When Defining an Employee

At least one of the default employees must be a valid Active Directory user to be able to log into the Portal for the first time.

If you are defining employees for the first time, be sure to create the employee definition for yourself as administrator. Use your Windows login name as your employee name. If you are creating a definition for another employee, use that employee’s Windows login name.

Employee definitions in the application are used as follows:

• As the logged-in employee to the Portal, the Organization default is your primary organization. Only the organizations in which you have a direct role or a role through propagation will be available for selection on the Portal pages.
• When performing a record search in the application, you can enter owner as part of the search criteria. The owner is the employee associated with the quality records to be searched.

Note the following:

• Employee contains these optional modeling definitions: Organization, Dictionary, ESig Role Group, Factory, Training Plan, Operation, and Work Center.
• Employee is an optional field in all container transactions and in all resource transactions.

When Defining Escalation Recipients

An escalation recipient is the person who will receive an e-mail notification when an employee has upcoming, due, or overdue work. For example, you can send an e-mail notification to an employee's manager if work is due. You define escalation recipients using the Employee field in the Escalation Recipients grid.

The Change Mgt Settings modeling object enables you to set up reminder notifications to be sent to escalation recipients when an approver or collaborator has an incomplete task. The application sends the reminder to the escalation recipient specified for the employee if you select this type of reminder notification. Refer to "Defining Change Mgt Settings" for information on setting the approval and collaboration reminders.

The application will not send e-mail notifications if you do not select escalation recipients in the Escalation Recipients grid. Ensure that the E-mail Address field is complete in each escalation recipient's Employee modeling object instance or the recipients will not receive the e-mail notifications.

Document Management

You can specify a user name and password to use to access documents stored in an external document repository. The Authentication Type field on the Document modeling object must be set to Basic to use the user name and password fields. Refer to "When Accessing Files in an External Repository" for information.

When Assigning Filter Tags to an Employee

You can use the Filter Tags Assignment section to control how the employee can use the filter tags feature and allow the employee's filter tag selections to persist from session to session. The employee has the option to further refine their filter tag selections using the Set Filter Tags pop-up. The Set Filter Tags pop-up is accessed by clicking Settings on the Primary Navigation bar and then clicking Set Filter Tags in the fly-out menu.

Important: As an administrator you must set your own filter tag assignment to All Instances before assigning filter tags to other employees. This ensures proper access to all tagged object instances.

There are three radio buttons in the Filter Tag Assignment section that control how the employee can use the filter tags feature.

• All Instances

Selecting All Instances allows the employee to view all instances, regardless of the filter tag status. The Assigned Filter Tags grid is hidden and any filter tags that were previously assigned to the employee are removed when the page is saved. The Set Filter Tag pop-up will be disabled.

• Tagged and Untagged

Selecting Tagged and Untagged allows the employee to view instances with filter tags that are assigned to them and instances that are not tagged. This is the default selection for new Employee object instances.

• Tagged Only

Selecting Tagged Only allows the employee to view only instances associated with the filter tags assigned to them.

When you assign filter tags using the Assigned Filter Tags grid, by default those filter tags are selected in the Set Filter Tag pop-up accessed from the Global Header. An employee can use the pop-up to deselect any filter tags not needed for that session. The application then deselects the filter tag in the Assigned Filter Tags grid. This allows the employee's selections to persist from session to session.

If an administrator creates or updates a filter tag for an instance, an employee will not have access to that instance unless the new or updated filter tag is added to the employee's Assigned Filter Tags grid or the employee is set to view All Instances. Additionally, if an employee assigns filter tags to a modeling object that the employee is not assigned to, any changes they attempt may not reflect changes being viewed; it is recommended for employees to first make sure that the administrator has assigned the filter tags to the employee before the employee attempts to assign them to a modeling object.

Note: When deleting a filter tag from the grid you must deselect all filter tags and then select only the tag you want to delete. After deleting the tag you can reselect the filter tags you want to assign to the employee.

How to Define an Employee

Follow these steps to define an Employee:

1. Open the Employee page. The Employee page appears within the Modeling page.
2. Click New. Blank fields appear for you to define a new instance.
3. Enter the employee's Windows login in the Employee field.
4. Enter optional information according to your business requirements. Refer to the field definitions table for information on the optional fields.
5. Click Save. The application saves the modeling object and displays a success message.

How to Assign One or More Roles to an Employee

Follow these steps to assign one or more roles to an employee.

1. Perform the "How to Define an Employee" procedure.

Or

Select an existing Employee instance.

2. Click Add new row in the Roles grid. A new row appears.
3. Select a Role.
4. Select the Organization for which the role is valid.
5. Select the Propagate check box to propagate the role to the specified organization’s sub-organizations if necessary.
6. Repeat steps 2-5 to add another role for this employee.
7. Click Save. The application displays a success message indicating the modeling object was updated.

How to Assign an Escalation Recipient to an Employee

Follow these steps to assign an escalation recipient to an employee:

1. Perform the "How to Define an Employee" procedure.

Or

Select an existing Employee instance.

2. Click Add new row in the Escalation Recipients grid. A new row appears.
3. Select the employee to receive escalation e-mail notifications from the Employee list.
4. Repeat steps 2-3 to add another escalation recipient.
5. Click Save. The application displays a success message indicating the modeling object was updated.

How to Assign Filter Tags to an Employee

Important: As an administrator you must set your own filter tag assignment to All Instances before assigning filter tags to other employees. This ensures proper access to all tagged object instances

Follow these steps to assign filter tags to an employee:

1. Perform the "How to Define an Employee" procedure.

Or

Select an existing Employee instance.

2. Select the desired radio button for the Filter Tag Assignment Type.
• The Assigned Filter Tags grid is hidden if you selected All Instances. Skip to step 4.
• Proceed to step 3 if you selected Tagged and Untagged or Tagged Only.
3. Select the filter tags you want to assign to the employee in the Assigned Filter Tag(s) grid.
4. Click Save. The application saves the modeling object and displays a success message.
