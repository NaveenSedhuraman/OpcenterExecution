Definieren von Nutzercodes

Nutzercodes sind Codelisten, die auf die Anforderungen jeder einzelnen Installation abgestimmt sind, und können wertvolle Verfolgungs- und Berichtsinformationen enthalten. Benutzerdefinierte Codes ermöglichen es Ihnen, eine Liste von zulässigen Werten für ein bestimmtes Feld anzugeben.

Die meisten Nutzercodes bestehen aus drei Attributen: Bezeichnung, Beschreibung und Anmerkungen. Einige Nutzercodes enthalten weitere Attribute für zusätzliche Funktionen oder zur Überprüfung. Für einige Nutzercodes können darüber hinaus auch zugehörige Umlaufbestandsnachrichten vorliegen, wenn sie mit einem Feld einer Containerdefinition verbunden sind.

Beim Definieren von Nutzercodes

Die Bezeichnung für jeden Nutzercode muss innerhalb des Typs eindeutig sein. Der Wert wird als alternativer Schlüssel für Suche, Dateneingabe und Überprüfung verwendet.

Sie können auch nach Belieben neue Nutzercodetypen erstellen.

Einzelheiten zu den Feldern, die alle Modellierungsobjekte aufweisen, finden Sie unter "Gemeinsame Felder auf Modellierungsseiten".

Nutzercodes

Diese Tabelle beschreibt die Standard-Nutzercodes. Beschreibungen der Nutzercodes, für die Umlaufbestandsnachrichten vorliegen können, finden Sie in der Beschreibung der nächsten Tabelle.

Nutzercode

Beschreibung

Stücklistentyp

Der Stücklistentyp wird verwendet, um den Zweck der Stücklistenstruktur zu unterscheiden. Es kann mehrere Stücklisten für dasselbe Produkt geben. Stücklisten können eine höhere oder niedrigere Auflösungsstufe haben, je nachdem, in welcher Abteilung die Stückliste verwendet wird. Elemente, die kein physischer Bestandteil des Fertigungsprozesses sind, können hinzugefügt und für die standardmäßigen Kostenberechnungen oder Arbeitsberichte verwendet werden.

"Stücklistentyp" ist ein optionales Feld in diesen Modellierungsdefinitionen:

• Stückliste
• ERP-Stückliste

Bonusgrund

Der Bonus ist eine Vergrößerung der Menge eines Containers und "Bonusgrund" ist ein optionales Feld in der Modellierungsdefinition "Bonusgrundgruppe". Bonusgründe werden vom Modellierungsobjekt "Operation" referenziert, jedoch nur dann, wenn eine Bonusgrundgruppe angegeben ist.

Kaufgrund

"Kaufen" bedeutet eine Erhöhung der Menge eines Containers, die in der Regel einer Buchungseinheit, wie etwa der Entwicklungsabteilung zugerechnet wird.

"Kaufgrund" ist ein optionales Feld in der Modellierungsdefinition "Kaufgrundgruppe". Kaufgründe werden vom Modellierungsobjekt "Operation" referenziert, jedoch nur dann, wenn eine Kaufgrundgruppe angegeben ist.

Behälterstatuscode

Status, der einer Behälterressource zunächst bei der Einrichtung und später bei jeder Statusänderung zugewiesen werden kann. Weitere Informationen finden Sie unter "Definieren von Behälterstatuscodes".

Behälterstatusgrund

Der Grund für den aktuellen Status des Behälters (zugewiesener Behälterstatuscode). Weitere Informationen finden Sie unter "Definieren von Behälterstatusgründen".

Ursachencode

Gibt die Hauptursache für einen Fehler an. Ursachencodes werden Qualitätsdatensätzen zugewiesen. Weitere Informationen finden Sie unter "Definieren von Ursachencodes".

Grund für Statusänderung

Der Ursachencode für die Änderung eines Containerstatus.

Klassifikation

Ein erforderlicher Indikator, der in Kombination mit dem Modellierungsobjekt "Unterklassifikation" verwendet wird, um den Qualitätsdatensatztyp festzulegen. Weitere Informationen finden Sie unter "Definieren von Klassifikationen".

Kommentartyp

Kategorien für Benutzerkommentare, die einem generischen oder Produktionsereignis hinzugefügt werden. Weitere Informationen finden Sie unter "Definieren von Kommentartypen".

Komponentendefektgrund

Ein Komponentendefekt ist eine defekte Stücklistenkomponente, die jedoch nicht als Verlust in Bezug auf die Menge des betreffenden Containers gewertet wird.

"Komponentendefektgrund" ist ein optionales Feld in der Modellierungsdefinition "Komponentendefektgrundgruppe". Komponentendefektgründe werden vom Modellierungsobjekt "Operation" referenziert, jedoch nur dann, wenn Sie eine Komponentendefektgrundgruppe definieren.

Containerdefektgrund

Der Containerdefektgrund ist der Grund, weshalb ein Container für defekt erklärt wird.

"Containerdefektgrund" ist ein optionales Feld in der Modellierungsdefinition "Containerdefektgrundgruppe". Containerdefektgründe werden vom Modellierungsobjekt "Operation" referenziert, jedoch nur dann, wenn Sie eine Komponentendefektgrundgruppe definieren.

Delegierungsgrund

Grund für die Neuzuweisung einer Aufgabe von einem Mitarbeiter an einen anderen. Weitere Informationen finden Sie unter "Definieren von Delegierungsgründen".

Elektr. Unterschrift: Mitunterzeichnergrund

Der Grund für eine elektronische Mitunterzeichnung. Weitere Informationen finden Sie unter "Definieren von Mitunterzeichnungsgründen".

Hinweis: Dieser Nutzercode ist nur im Modul "Electronic Signatures" gültig.

Elektr. Unterschrift: Bedeutung

Gibt Zweck und Verantwortlichkeit einer Unterschrift an. Weitere Informationen finden Sie unter "Definieren von Bedeutungen für elektronische Unterschriften".

Hinweis: Dieser Nutzercode ist nur im Modul "Electronic Signatures" gültig.

Fehleraktionstyp

Eine Aktion, die als Antwort auf eine bestimmte Ursache durchgeführt werden kann. Weitere Informationen finden Sie unter "Definieren von Fehleraktionstypen".

Fehlermodus

Die tatsächliche Ursache eines Fehlers. Weitere Informationen finden Sie unter "Definieren von Fehlermodi".

Fehlerschweregrad

Schweregrad eines Fehlers während eines Ereignisses. Weitere Informationen finden Sie unter "Definieren von Fehlerschweregraden".

Fehlertyp

Charakteristisches Merkmal des Fehlers. Ermöglicht die Kategorisierung von Fehlern. Weitere Informationen finden Sie unter "Definieren von Fehlertypen".

Filter-Tag

Tag, das Modellierungsobjektinstanzen zugewiesen wird, um die Datenfilterung zu erleichtern.

Verbrauchsabweichungsgrund

Der Grund für die Differenz zwischen der erforderlichen und der tatsächlich verbrauchten Menge.

Verbrauchsgrund

Zweck des Komponentenverbrauchs.

Lokale Nacharbeit Grund

Implementierungen von Opcenter EX MDD und Opcenter EX CR verwenden dieses Objekt nicht standardmäßig. Das Objekt steht Ihnen jedoch in Ihrer benutzerdefinierten Implementierung zur Verfügung.

Verlustgrund

Als Verluste werden während der Bearbeitung verloren gegangene Containereinheiten bezeichnet. "Verlustgrund" ist ein optionales Feld in der Modellierungsdefinition "Verlustgrundgruppe". Verlustgründe werden vom Modellierungsobjekt "Operation" referenziert, jedoch nur dann, wenn Sie eine Verlustgrundgruppe definieren.

"Verlustgrund" ist ein erforderliches Feld in der Transaktion "Mengenänderung", wenn die ausgewählte Aktion "Verlust" ist.

Wartungsgrund

Der Grund für die Wartung der Ressource.

"Wartungsgrund" ist ein erforderliches Feld in den folgenden Modellierungsdefinitionen:

• Datumsanforderung
• Wiederholungsterminanforderung
• Durchsatzanforderung

Hinweis: Der Nutzercode ist nur für das Modul "Maintenance Management" gültig.

Nachrichtenkategorie

Beschriftung zur Identifizierung der Nachrichtentypen, die in Concierge und in der Nachrichtenzentrale angezeigt werden. Weitere Informationen finden Sie unter "Definieren von Nachrichtenkategorien".

"Nachrichtenkategorie" ist ein erforderliches Feld im Modellierungsobjekt "Portalnachrichtenkategorie".

NCR-Fehlercode

Implementierungen von Opcenter EX MDD und Opcenter EX CR verwenden dieses Objekt nicht. Das Objekt steht Ihnen jedoch in Ihrer benutzerdefinierten Implementierung zur Verfügung.

NCR-Lösungscode

Implementierungen von Opcenter EX MDD und Opcenter EX CR verwenden dieses Objekt nicht. Das Objekt steht Ihnen jedoch in Ihrer benutzerdefinierten Implementierung zur Verfügung.

Tätigkeit

Implementierungen von Opcenter EX MDD und Opcenter EX CR verwenden dieses Objekt nicht. Das Objekt steht Ihnen jedoch in Ihrer benutzerdefinierten Implementierung zur Verfügung.

Auftragsstatus

ERP-Produktionsauftragsstatus. Mögliche Werte sind "Freigegeben", "Beendet", "In Bearbeitung", "Geschlossen" und "Abgebrochen". Informationen zum Definieren von Auftragsstatus für die Verwendung mit Opcenter Execution Core Scheduling finden Sie unter "Definieren von Auftragsstatus".

Auftragstyp

ERP-Produktionsauftragstyp. Hier können mehrere verschiedene Werte angezeigt werden, z. B.: "Standard", "Nacharbeit" oder "Testlauf".

Paketerstellungsgrund

Der Grund für die Erstellung eines Änderungsverwaltungspakets. Weitere Informationen finden Sie unter "Definieren von Nutzercodes für die Änderungsverwaltung".

Paketprioritätscode

Ein Indikator, der verwendet wird, um einem Paket eine Priorität zuzuweisen. Weitere Informationen finden Sie unter "Definieren von Nutzercodes für die Änderungsverwaltung".

Pakettyp

Verwendeter Indikator, um den Typ eines Änderungsverwaltungspakets anzugeben. Weitere Informationen finden Sie unter "Definieren von Nutzercodes für die Änderungsverwaltung".

Prioritätsstufe

Ein Indikator, der verwendet wird, um einem Qualitätsdatensatz eine Bearbeitungspriorität zuzuweisen. Weitere Informationen finden Sie unter "Definieren von Prioritätsstufen".

Prozesstimertyp

Indikator für die Gruppierung, Filterung und Berichterstellung zu Prozesstimern. Weitere Informationen finden Sie unter "Definieren von Prozesstimertypen".

Mengenanpassungsgrund

Mengenanpassungsgründe werden in der Regel zur Beschreibung von Anpassungen nach oben oder unten verwendet, die nicht mit den Bearbeitungsaktivitäten im Zusammenhang stehen. Sie können die Menge eines Containers z. B. nach Beendigung der jährlichen Inventur anpassen.

"Mengenanpassungsgrund" ist ein optionales Feld in der Modellierungsdefinition "Mengenanpassungsgrundgruppe". Mengenanpassungsgründe werden vom Modellierungsobjekt "Operation" referenziert, aber nur dann, wenn Sie eine Mengenanpassungsgrundgruppe angeben.

Qualitätsdatensatzauflösungscode

Grund für die Lösung (Schließung) eines Qualitätsdatensatzes. Weitere Informationen finden Sie unter "Definieren von Auflösungscodes für Qualitätsdatensätze".

Freigabegrund

Der Grund, weshalb ein Container aus dem Status "Angehalten" freigeben wird.

Entnahmegrund

Der Grund, weshalb Material aus einem Umlaufbestands-Container entnommen wurde.

Entnahmeunterschiedsgrund

Der Grund, weshalb die verbrauchte Menge u. U. von der tatsächlich entnommenen Menge abweicht.

Ersetzungsgrund

Grund für die Ersetzung von Komponenten.

Mithilfe des Kontrollkästchens für das Ausschussteil, das ersetzt wird, können Sie eine Komponente, die bei einem Austausch entfernt wird, als Ausschuss festlegen.

Ressourcenstatuscode

Der Status, der einer Ressource zunächst bei der Einrichtung und später bei jeder Statusänderung zugewiesen werden kann. Weitere Informationen finden Sie unter "Definieren von Ressourcenstatuscodes".

Dieser Wert wird für Abfragen und zur Erstellung von Berichten verwendet. (Mittlere Zeit zwischen 2 Fehlern (MTBF), Mittlere Zeit bis eine Reparatur erforderlich wird (MTTR), Zeit im aktuellen Zustand (TAS) usw.)

Ressourcenstatusursache

Die Ursache für den aktuellen Status einer Ressource (zugewiesener Ressourcenstatuscode). Weitere Informationen finden Sie unter "Definieren von Ressourcenstatusursachen".

Ressourcentyp

Eine Beschreibung zur Identifizierung eines Ausrüstungstyps. Weitere Informationen finden Sie unter "Definieren von Ressourcentypen".

Rollupgrund

Änderungen der Containermenge werden auf dem untersten Containerlevel (Kind-Container) erfasst und können mehrere Ursachencodes aufweisen. Die Mengen werden dann auf dem mittleren und obersten Eltern-Containerlevel zusammengefasst. Bei dieser Zusammenfassung wird den Eltern-Containern nur ein Ursachencode zugewiesen. Dies ist der Rollupgrund.

"Rollupgrund" ist ein optionales Feld in der Modellierungsdefinition "Rollupgrundgruppe". Rollupgründe werden vom Modellierungsobjekt "Operation" referenziert, jedoch nur dann, wenn Sie eine Rollupgrundgruppe definieren.

Ausschussgrund

Grund, der angegeben werden kann, wenn ein Container als Ausschuss eingestuft wird.

Verkaufsgrund

Grund, weshalb ein interner Transfer von Containern aufgezeichnet wird, der keinen negativen Einfluss auf die Ausbeute hat. Der Verkaufsgrund ist der für einen solchen Transfer angegebene Grund; für ein Los könnte dieser Grund z. B. lauten: "An Entwicklung zum Testen eines neuen Prozesses verkauft".

"Verkaufsgrund" ist ein optionales Feld in der Modellierungsdefinition "Verkaufsgrundgruppe". Verkaufsgründe werden vom Modellierungsobjekt "Operation" referenziert, jedoch nur dann, wenn Sie eine Verkaufsgrundgruppe definieren.

Unterklassifikation

Ein erforderlicher Indikator, der in Kombination mit dem Modellierungsobjekt "Klassifikation" verwendet wird, um den Qualitätsdatensatztyp festzulegen. Weitere Informationen finden Sie unter "Definieren von Unterklassifikationen".

Ersetzungsgrund

Der Grund, weshalb nicht das vorgesehene, sondern ein anderes Produkt eingesetzt wurde.

Trainingseintragsstatus

Ein Status der angibt, ob der Mitarbeiter für die Ausführung der Aufgabe qualifiziert ist. Weitere Informationen finden Sie unter "Definieren von Trainingseintragsstatus".

Nutzercodes mit Umlaufbestandsnachrichten

Diese Tabelle beschreibt die Nutzercodes, für die zugehörige Umlaufbestandsnachrichten vorliegen können.

Nutzercode

Beschreibung

Anhaltegrund

Wenn Sie einem Container einen Anhaltegrund zuweisen, wird verhindert, dass Transaktionen für den betreffenden Container durchgeführt werden. Anhaltegrundcodes werden vom Benutzer definiert und können mehrere Beschreibungen (Gründe) für das Anhalten umfassen.

Der "Anhaltegrund" ist ein erforderliches Feld der Transaktionen "Anhalten" und "Anhalten (mehrere)". Ein Anhaltegrund ist erforderlich, wenn Sie ein Produktionsereignis aufzeichnen und der Benutzer angibt, dass die betreffenden Container angehalten werden sollen.

"Anhaltegrund" ist ein optionales Feld bei folgenden Transaktionen:

• Containersuche
• Freigeben

Eigentümer

Jedem Container ist ein hat einen zugeordneten Eigentümercode zugewiesen. Der Eigentümercode wird zur Gruppierung verwendet, um Container zur Abfrage und zur Bearbeitung zu trennen. Beispiele für Nutzercode sind Produktion, Entwicklung, Prototypen, Verkaufsproben usw.

"Eigentümer" ist ein erforderliches Feld bei den Transaktionen "Start" und "Starten - zwei Level".

"Eigentümer" ist ein optionales Feld in der Transaktion "Containerwartung".

Prioritätscode

Eine Instanz eines Prioritätscodes ist eine Methode, um einem Container eine Bearbeitungspriorität zuzuweisen. Jeder Prioritätscode enthält eine Beschreibung und einen relativen Prioritätswert. Der relative Prioritätswert wird verwendet, um die Reihenfolge festzulegen, in der die Container erledigt werden (unabhängig von Codename oder Beschreibung). Prioritätscodes sind nur einer der Algorithmen, die verwendet werden, um einen numerischen Wert für die Ablaufplanung und Erledigung zu bestimmen.

"Prioritätscode" ist ein optionales Feld in der Transaktion "Container".

Produkttyp

Produkttypcodes werden verwendet, um zwischen verschiedenen Produkttypen wie Umlaufbestand, Rohmaterial, Fertigen Erzeugnissen usw. zu unterscheiden.

"Produkttyp" ist ein erforderliches Feld in der Produktdefinition.

Nacharbeitsgrund

Eine Instanz des Nacharbeitsgrunds wird genutzt, um eine Liste von Nacharbeitsgründen zu liefern. Der Code wird einem Container zugewiesen, wenn dieser in eine Nacharbeitsschleife eintritt. Er wird dann allen Durchsatztransaktionen zugewiesen, die vom Container erzeugt werden, während dieser sich in der Nacharbeitsschleife befindet. Der Nacharbeitsgrund bietet daher z. B. die Möglichkeit, in Berichten zwischen erstem Durchgang und Nacharbeit zu unterscheiden.

"Nacharbeitsgrund" ist ein optionales Feld in der Definition von:

• Operation
• Nacharbeitsgrundgruppe

"Nacharbeitsgrund" ist ein erforderliches Feld in diesen Transaktionen:

• Nacharbeit
• Transfer zur Nacharbeit

Versandgrund

Versandgründe werden den Versandinformationen des Containers zugewiesen.

Startgrund

Jeder Container verfügt über einen zugewiesenen Startgrundcode. Startgrundcodes sind als Auswahlkriterien bei Abfragen des Umlaufbestandsstatus und bei der Transaktionsberichterstattung (basierend auf der Transaktionshistorie) verfügbar.

"Startgrund" ist ein erforderliches Feld in den Transaktionen "Start" und "Starten - zwei Level".

"Startgrund" ist ein optionales Feld in der Transaktion "Containerwartung".

Maßeinheit

Die verwendete Maßeinheit.

Maßeinheit für die Containermenge. Weitere Informationen finden Sie unter "Definieren von Maßeinheiten".

"Maßeinheit" ist ein optionales Feld in folgenden Definitionen:

• Produkt
• Produktfamilie
• Produktionsauftrag
• Verkaufsauftrag

"Maßeinheit" ist ein optionales Feld in folgenden Transaktionen:

• Containerwartung
• Ressourcendurchsatz
• Start
• Starten - zwei Level

So wird ein Nutzercode definiert

Gehen Sie folgendermaßen vor, um einen Nutzercode zu definieren:

1. Öffnen Sie die Seite Nutzercodegruppe. Die Seite "Nutzercodegruppe" wird innerhalb der Seite "Modellierung" angezeigt.
2. Klicken Sie auf Neu. Es werden leere Felder zum Definieren einer neuen Instanz angezeigt.
3. Geben Sie einen Namen für den Nutzercode in das Feld Name ein.
4. Geben Sie abhängig von den jeweiligen Geschäftsanforderungen optionale Informationen ein. Unter "Gemeinsame Felder auf Modellierungsseiten" finden Sie Informationen zu den optionalen Feldern.
5. Klicken Sie auf Speichern. Die Anwendung speichert das Modellierungsobjekt und zeigt eine Erfolgsmeldung an.
