Informationen im Vorfeld der Implementierung

Bevor Sie die Wartungsverwaltung in Ihrer Fabrik implementieren, müssen Sie mit den Typen von Wartungsanforderungsplänen vertraut sein und einige Entscheidungen zur Konfiguration der Wartungsverwaltung treffen:

• Welche Ressourcen sollen in die geplante Wartung einbezogen werden?
• Welche Wartungsklassen möchten Sie definieren und welche Ressourcen werden ihnen zugewiesen?
• Wie lauten die Wartungsgründe der Wartungsanforderungstypen?
• Welches sind die auf Durchsatz, Einzeldatum bzw. Wiederholungstermin basierenden Wartungsanforderungspläne? Welcher Plan oder welche Pläne sollen für eine Ressource gelten?
• Wer soll die E-Mail-Benachrichtigungen über ausstehende, fällige und überfällige Wartungen erhalten?
• Zu welchem Zeitpunkt soll die Aktivierung und Durchsetzung der Wartungsanforderungen beginnen?


Wartungsanforderungspläne

Eine Wartungsanforderung hängt vom Typ des festgelegten Zeitplans ab, den Sie für die zugehörigen Ressourcen der Anforderung erstellen möchten:

• Maximaler Durchsatz, den eine Ressource bearbeiten kann
• Einzelnes Kalenderdatum
• Folge von Wiederholungsterminen

Eine Ressource kann mehr als einen Anforderungsplan haben.

Planung nach Durchsatz

Eine Durchsatzanforderung definiert die maximale Stückzahl, die die Ressource zwischen zwei Wartungen bearbeiten kann, bevor die nächste Wartung fällig ist. Durchsatzanforderungspläne beinhalten Toleranzbereiche für eine Toleranzmenge und eine Warnmenge.

Planung nach Datum

Eine Datumsanforderung legt ein bestimmtes Datum fest, an dem die nächste Wartung der Ressource stattfinden soll. Datumsanforderungspläne enthalten Toleranz- und Warnbereiche.

Planung nach Wiederholungstermin

Eine Wiederholungsterminanforderung definiert den Wartungsplan einer Ressource basierend auf wiederkehrenden Daten im Kalender. Der Zeitplan gibt ein Datum oder mehrere Daten an, an denen eine Wartung durchgeführt werden muss. Jede Wartung muss unabhängig davon durchgeführt werden, wann die vorhergehende Wartung ausgeführt wurde. Wenn z. B. für jeden zweiten Dienstag eine Wartung geplant ist, sie aber an einem Donnerstag und damit zwei Tage zu spät durchgeführt wurde, wird die nächste Wartung dennoch für einen Dienstag geplant.

Eine Wiederholungsterminanforderung wird auf ähnliche Weise eingerichtet wie ein Wiederholungstermin in Microsoft Outlook.

Wiederholungsterminanforderungs-Pläne enthalten Toleranz- und Warnbereiche.

Enddatum vs. offenes Ende

Eine Wiederholungsterminanforderung kann ein Enddatum haben oder offen sein. Sie können ein genaues Enddatum oder die Anzahl von Wiederholungen festlegen. Wenn Sie die Anzahl von Wiederholungen angeben, wird diese Zahl verwendet, um das Enddatum zu berechnen. Die Anforderung beinhaltet auch einen ersten Termin, der dem Datum entspricht, an dem der Zeitplan erstellt wird.

Muster und Häufigkeit

Bei den Planungsmustern für die Wartung werden folgende Intervalle unterschieden:

• Einmalig – Die Pläne werden einmalig zum festgelegten Datum und zur festgelegten Uhrzeit ausgeführt.
• Sekunden – Zwischen den Ausführungen der Pläne liegt eine festgelegte Anzahl von Sekunden.
• Minuten – Zwischen den Ausführungen der Pläne liegt eine festgelegte Anzahl von Minuten.
• Stündlich – Zwischen den Ausführungen der Pläne liegt eine festgelegte Anzahl von Stunden.
• Täglich – Zwischen den Ausführungen der Pläne liegt eine festgelegte Anzahl von Tagen.
• Wöchentlich – In den Zeitplänen ist ein bestimmter Wochentag festgelegt.
• Monatlich – In den Zeitplänen ist ein bestimmter Tag des Monats festgelegt.
• Jährlich – In den Zeitplänen sind ein Monat und ein Tag dieses Monats festgelegt.

Allen diesen Intervallen kann eine bestimmte Häufigkeit zugewiesen werden, aus der sich ein Muster dafür ergibt, wie oft die Anforderung wiederholt wird: 1 bedeutet, dass die Anwendung die Wartung bei jeder geplanten Wiederholung ausführt; 2 bedeutet bei jeder zweiten geplanten Wiederholung usw.

Systemberechnung für Datumswerte

Datumwerte sind Zeitstempel, die in der Anwendung gesetzt werden. Sie werden entweder von Ihnen eingegeben oder von der Anwendung als Standardwert vorgegeben. Wenn Sie bei der Definition von Datumsanforderungen oder Wiederholungsterminanforderungen keine Uhrzeit eingeben, wird der Standardwert Mitternacht verwendet.

Bei einem Zeitplan mit Wiederholungsterminen speichert die Anwendung nicht alle Datumswerte. Sie speichert nur das Datum der nächsten geplanten Wartung. Für Zeitpläne, die auf einen Tag fallen, den es im nächsten Monat nicht gibt, verwendet die Anwendung den letzten Tag des Monats.

Wenn eine Wiederholungsterminanforderung erstellt wird oder eine Änderung der Anforderung den Zeitplan beeinflusst, berechnet die Anwendung das nächste zukünftige Datum im Zeitplan und speichert das Datum mit der Anforderung.

Mehrere Zeitpläne

Wie bereits erwähnt, kann eine Ressource mehr als einen Typ von Wartungsplan erfordern.

Beispiel 1

Eine Ressource muss regelmäßig auf wöchentlicher und vierteljährlicher Basis gewartet werden. Deshalb definieren Sie zwei Wiederholungsterminanforderungen, eine auf wöchentlicher und eine auf vierteljährlicher Basis:

Die wöchentliche Wartung soll jeden Donnerstag stattfinden. Deshalb ist der Zeitplan "Wöchentlich", der Tag ist "Donnerstag" und die Häufigkeit ist "1".

Die vierteljährliche Wartung soll vierteljährlich am 9. des Monats erfolgen. Deshalb ist der Zeitplan "Monatlich", der Tag des Monats ist "9" und die Häufigkeit ist "3" (einmal alle drei Monate).

Beispiel 2

Eine Ressource muss nach einer bestimmten Durchsatzmenge und periodisch auf jährlicher Basis gewartet werden. Deshalb müssen Sie zwei Pläne erstellen: Eine Durchsatzanforderung, die auf der Durchsatzmenge basiert, und eine Wiederholungsterminanforderung mit jährlicher Wiederholung.

E-Mail-Benachrichtigungen

Wenn Sie die Wartungsanforderungspläne festlegen, bestimmen Sie, wer entsprechend den von Ihnen festgelegten Warn- und Toleranzstufen per E-Mail über ausstehende, fällige und überfällige Wartungen informiert wird. Diese E-Mail-Benachrichtigungen werden nicht automatisch erstellt, sobald eine Wartung fällig wird, sondern erst dann, wenn der Bediener eine Transaktion an der Ressource ausführt, etwa "Anmelden" oder "Ressourcendurchsatz".

Innerhalb eines Wartungszyklus wird eine E-Mail, mit Ausnahme der "Überfällig"-Benachrichtigung, nur einmal versandt. "Überfällig"-Benachrichtigungen werden jedes Mal versandt, wenn eine Transaktion an der Ressource ausgeführt wird und die Toleranzkriterien überprüft wurden.

Wichtig: Bei einer Durchsatzanforderung wird die erste Fertigungstransaktion, mit der eine überfällige Ressource das Limit überschreitet, noch ausgeführt. Auch die "Überfällig"-Benachrichtigung wird versandt. Bei allen weiteren Transaktionen, die an der Ressource ausgeführt werden, wird die "Überfällig"-Benachrichtigung versandt, sie schlagen jedoch fehl.

Modellierungsdefinitionen

Dies sind die Modellierungsobjekt-Instanzen, die Sie definieren und ändern, um die Wartungsverwaltung zu implementieren.

Modellierungsobjekt

Beschreibung

Wartungsgrund

Der Grund für eine Wartungsanforderung.

Durchsatzanforderung

Die maximal zulässige Menge, die bearbeitet werden kann, bevor eine Wartung stattfinden muss. Bezieht sich auf Wartungsanforderungspläne, die auf Durchsatzmengen basieren.

Datumsanforderung

Ein bestimmtes Datum, an dem eine Wartung durchgeführt werden muss. Bezieht sich auf Wartungspläne, die auf einem einzelnen Datum basieren.

Wiederholungsterminanforderung

Ein oder mehrere Daten, an denen eine Wartung durchgeführt werden muss. Bezieht sich auf Wartungspläne, die auf Wiederholungsterminen basieren.

Wartungsklasse

Wartungsklassifikation – wird für Ressourcen mit derselben Wartungsanforderung verwendet. Eine Wartungsklasse kann viele Ressourcen enthalten, aber eine Ressourcen kann nur eine Wartungsklasse referenzieren.

Eine Ressource kann ein oder mehrere aktive Wartungsanforderungen haben, ohne zu einer Wartungsklasse zu gehören.

Ressource

Maschinen oder Ausrüstungsteile, für die eine aktive Anforderung erzwungen wird. Sie kann, muss aber nicht zu einer Wartungsklasse gehören.

Vorgehensweise zum Erzwingen von Anforderungen

Eine Wartungsanforderung wird nur dann aktiviert und eine Wartung wird nur dann erzwungen, wenn sie einer Ressource oder Wartungsklasse zugewiesen ist. Dies geschieht entweder über die Transaktion "Ressourcenaktivierung" oder über die Transaktion "Wartungsklassenaktivierung". Weitere Informationen finden Sie im Opcenter Execution Medical Device and Diagnostics Shop Floor-Benutzerhandbuch oder im Opcenter Execution Core Shop Floor-Benutzerhandbuch.

Der Wartungsplan einer Ressource enthält nur aktive, momentan zugewiesene Wartungsanforderungen.
