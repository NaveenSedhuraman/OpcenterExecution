Definieren von Operationen

Eine Operation ist ein Produktions- oder Bearbeitungspunkt, an dem Lager- und Produktionsaktivitäten verfolgt werden. Eine Operation wird von einer Spezifikation eines Arbeitsplanschritts referenziert.

In der Operationsdefinition werden Elemente wie Ursachencodes, Bearbeitungsregeln und die erlaubten Transaktionen für den Transport des Materials im Verlauf der Operation beschrieben. Im Gegensatz werden in Spezifikationen die Verarbeitungsschritte definiert, die bei einer Operation durchgeführt werden.

Hier einige Beispiele für Operationsinformationen:

• Referenzen auf Nutzercodes für die Aufzeichnung von Verlust-, Bonus-, Nacharbeits- und Versandinformationen
• Optionale Kennzeichnung der Operation als "Bestand", "Unterwegs" oder "Externer Servicepunkt"
• Ausweisung zulässiger und unzulässiger Transaktionen
• Angabe, ob der Status "In Warteschlange" verwendet wird und daher eine Anmeldung erforderlich ist

Der Portal-Benutzer kann beim Erstellen eines Ereignisses eine Operation angeben, um den Produktions- oder Bearbeitungspunkt anzugeben, an dem das Ereignis festgestellt wurde.

Beim Definieren einer Operation

"Operation" enthält:

• Die erforderliche Modellierungsdefinition "Containerlevel" (eingegeben als "Durchsatzberichtsebene")
• Die optionalen Modellierungsdefinitionen "Erledigungsregel", "Versandziel", "Trainingsanforderungsgruppe", "Druckwarteschlange", "Arbeitsbereich" und "Nutzercodegruppen" (verschiedene Ursachencodegruppen)

"Operation" ist:

• Ein erforderliches Feld in der Definition einer Spezifikation
• Ein optionales Feld in der Definition eines Mitarbeiters
• Ein optionales Feld in den Transaktionen "Anmelden", "Abmelden - standardmäßig", "Abmelden - nicht standardmäßig", "Durchsatz", "Transfer", "Nacharbeit", "Transfer zur Nacharbeit", "Anhalten", "Freigeben", "Containerwartung", "Mengenänderung", "Containerdefekt", "Komponentendefekt", "Komponentenverbrauch" und "Komponentenentnahme"

Die Tabelle "Container in Warteschlange" wird u. U. nicht auf der Seite "Operative Ansicht" angezeigt, wenn "Warteschlange verwenden" nicht aktiviert wurde. Ausschlaggebend dafür sind die in der Linienzuordnung ausgewählte Operation und der ausgewählte Container. Weitere Informationen finden Sie im Opcenter Execution Medical Device and Diagnostics Shop Floor-Benutzerhandbuch oder im Opcenter Execution Core Shop Floor-Benutzerhandbuch.

Wählen Sie im Abschnitt "Bereitstellungsdetails" die Namen der Nutzercodegruppen aus, die den Gründen für Mengenänderungen und Defekte zugewiesen sind.

Das Kontrollkästchen "Eingehenden abweichenden Container ablehnen" ist nur dann gültig, wenn Ihr Unternehmen das Modul "Nonconformance Management" erworben hat. Wenn Sie dieses Kontrollkästchen aktivieren, verhindert die Anwendung, dass Benutzer einen Container mit offenen Abweichungen zur Operation bewegen.

Hinweis: Für eine Operation können Umlaufbestandsnachrichten vorliegen, wenn sie einem Feld einer Containerdefinition zugewiesen ist. Weitere Informationen finden Sie unter "Definieren von Umlaufbestandsnachrichten" sowie im Opcenter Execution Medical Device and Diagnostics Shop Floor-Benutzerhandbuch und im Opcenter Execution Core Shop Floor-Benutzerhandbuch.

Felddefinitionen der Seite "Operation"

In dieser Tabelle sind die Felder definiert, die nur auf der Seite "Operation" zu finden sind.

Siehe auch "Gemeinsame Felder auf Modellierungsseiten". Dieser Abschnitt enthält Informationen zu den Feldern, die alle Modellierungsobjekte aufweisen.

Feld

Definition

Typ

Allgemein

Arbeitsbereich

Bereich der Fertigung, in dem ähnliche Arbeiten durchgeführt werden. Z. B. könnte ein Bereich, in dem verschiedene Aktivitäten zur Verpackung von Produkten durchgeführt werden, als ein Arbeitsbereich definiert werden.

Optional

Bereitstellungsdetails

Mengenänderungsgründe

Verlustgründe

Name der Nutzercodegruppe, in der die Gründe für Verluste bei der Operation definiert sind. Als Verluste werden während der Bearbeitung verloren gegangene Containereinheiten bezeichnet.

Sie können die im System vorhandene Nutzercodegruppe "Verlustgrundgruppe" verwenden oder eine neue Gruppe für operationsspezifische Verlustgründe erstellen.

Optional

Verkaufsgründe

Name der Nutzercodegruppe, in der die Verkaufsgründe für die Operation definiert sind. Ein Verkauf ist eine Verringerung der Menge eines Containers. Er wird in der Regel einer Buchungseinheit zugerechnet und durch eine Kontonummer identifiziert.

Sie können die im System vorhandene Nutzercodegruppe "Verkaufsgrundgruppe" verwenden oder eine neue Gruppe für operationsspezifische Verkaufsgründe erstellen.

Optional

Mengenanpassungsgründe

Name der Nutzercodegruppe, in der die Gründe für die Änderung der Menge eines Containers bei der Operation definiert sind. Mengenanpassungsgründe werden in der Regel zur Beschreibung von Anpassungen nach oben oder unten verwendet, die nicht mit den Bearbeitungsaktivitäten im Zusammenhang stehen. Sie können die Menge eines Containers z. B. nach Beendigung der jährlichen Inventur anpassen.

Sie können die im System verfügbare Nutzercodegruppe "Mengenanpassungsgrundgruppe" verwenden oder eine neue Gruppe für operationsspezifische Mengenanpassungsgründe erstellen.

Optional

Bonusgründe

Name der Nutzercodegruppe, in der die Bonusgründe für die Operation definiert sind. Ein Bonus ist eine Erhöhung der Menge eines Containers.

Sie können die im System vorhandene Nutzercodegruppe "Bonusgrundgruppe" verwenden oder eine neue Gruppe für operationsspezifische Bonusgründe erstellen.

Optional

Kaufgründe

Name der Nutzercodegruppe, in der die Kaufgründe für die Operation definiert sind. Ein Kauf bedeutet eine Erhöhung der Menge eines Containers, die in der Regel einer Buchungseinheit, wie etwa der Entwicklungsabteilung zugerechnet wird.

Sie können die mitgelieferte Nutzercodegruppe "Kaufgrundgruppe" verwenden oder eine neue Gruppe für operationsspezifische Kaufgründe erstellen.

Optional

Standardrollupgrund

Name der Nutzercodegruppe, in der die Rollupgründe für die Operation definiert sind. Änderungen der Containermenge werden auf dem untersten Containerlevel (Kind-Container) erfasst und können mehrere Ursachencodes aufweisen. Die Mengen werden dann auf dem mittleren und obersten Eltern-Containerlevel zusammengefasst. Bei dieser Zusammenfassung wird den Eltern-Containern nur ein Ursachencode zugewiesen. Die Anwendung verwendet den in der Operation referenzierten Rollupgrund, es sei denn, dieser wird vom Benutzer bei der Mengentransaktion überschrieben.

Sie können die im System vorhandene Nutzercodegruppe "Rollupgrundgruppe" verwenden oder eine neue Gruppe für operationsspezifische Rollupgründe erstellen.

Optional

Defektgründe

Komponentendefektgründe

Name der Nutzercodegruppe, in der die Komponentendefektgründe für die Operation definiert sind. Ein Komponentendefekt ist eine defekte Stücklistenkomponente, die jedoch nicht als Verlust in Bezug auf die Menge des betreffenden Containers gewertet wird.

Sie können die im System vorhandene Nutzercodegruppe "Komponentendefektgrundgruppe" verwenden oder eine neue Gruppe für die operationsspezifischen Komponentendefektgründe erstellen.

Optional

Containerdefektgründe

Name der Nutzercodegruppe, in der die Containerdefektgründe für die Operation definiert sind. Ein Containerdefekt ist eine Containereinheit, die defekt ist, aber nicht als Verlust gezählt wird.

Sie können die im System vorhandene Nutzercodegruppe "Containerdefektgrundgruppe" verwenden oder eine neue Gruppe für die operationsspezifischen Defektgründe erstellen.

Optional

Defektgründe

Defektgrundgruppe, die die für die Operation verfügbaren Defektgründe enthält, wenn ein Container an der Operation im Popup "Defekt protokollieren" des Opcenter Execution Electronics-Produktions-Clients angezeigt wird.

Hinweis: Dieses Feld wird nur angezeigt, wenn Opcenter Execution Electronics installiert ist. Weitere Informationen finden Sie im Opcenter Execution Electronics-Benutzerhandbuch.

Optional

Nacharbeitsgründe

Nacharbeitsgründe

Name der Nutzercodegruppe, in der die Nacharbeitsgründe für die Operation definiert sind.

Sie können die im System vorhandene Nutzercodegruppe "Nacharbeitsgrundgruppe" verwenden oder eine neue Gruppe für die operationsspezifischen Nacharbeitsgründe erstellen.

Optional

Details

Versandziele

Named Object Group, die eine Gruppe von Versandzielen darstellt. Ein Versandziel ist ein Named Data Object, das die Fabrik oder den Kunden angibt, an die bzw. den der Container versendet wird. Wird verwendet, um eine Operation mit einem bestimmten Versandziel einzurichten. Beispielsweise muss die Unterbaugruppen-Operation in einer Fabrik ihre Produkte an ein bestimmtes Subunternehmen schicken, und die Endmontage-Operation schickt ihre Fertigwaren an ihren Kunden (das Versandziel).

Optional

Erledigungsregel

Liste von Erledigungsregelinstanzen. Verwenden Sie dieses Feld, um eine bestimmte Erledigungsregel für den Arbeitsbereich zu referenzieren.

Optional

Trainingsanforderungsgruppe

Die Trainingsgruppe, die zur Autorisierung eines Benutzers für die Durchführung einer bestimmten Aufgabe erforderlich ist.

Optional

Druckwarteschlange

Name des Netzwerkdruckers, der zum Ausdrucken von Containeretiketten im Zusammenhang mit der Operation verwendet wird.

Optional

Attribut

Unterwegs

Bei Aktivierung dieses Attributs wird angenommen, dass die Container an der Operation in Bezug auf die Anmeldung und die Verfolgung zwischen Fabriken und Subunternehmern den Status "Unterwegs" haben. Wenn dieses Attribut aktiviert ist, haben die Container den Status "Unterwegs".

Optional

Lagerort

Wenn dieses Attribut aktiviert ist, ist die Operation ein Lagerort. Wenn aktiviert, ist die Operation ein Lagerort. Eine Lageroperation kann ein Schritt eines Arbeitsplans oder eine eigenständige Operation sein.

Optional

Externer Servicepunkt

Wenn dieses Attribut aktiviert ist, handelt es sich bei der Operation um einen externen Servicepunkt, z. B. ein Subunternehmen. Wenn aktiviert, ist die Operation ein externer Servicepunkt. Diese Informationen werden zum Sortieren operationsbezogener Informationen und zur Erstellung entsprechender Berichte verwendet.

Optional

In Verarbeitung

Warteschlange verwenden

Wenn diese Option aktiviert ist, verfolgt die Operation die Container in der Warteschlange (die auf die Bearbeitung warten) anhand eines Warteschlangenstatus. Wenn aktiviert, verwendet die Operation einen Warteschlangenstatus. Bei Verwendung einer Warteschlange wird vor der Bearbeitung des Containers die Transaktion "Anmelden" durchgeführt, um den Status von "In Warteschlange" in "In Bearbeitung" zu ändern.

Optional

Durchsatzzusammenfassung

Wenn diese Option aktiviert ist, werden die Durchsatzinformationen auf dem obersten Containerlevel erfasst. Der Eintrag im Feld "Durchsatzberichtsebene" bleibt dabei unberücksichtigt.

Optional

Eingehenden abweichenden Container ablehnen

Wenn diese Option aktiviert ist, können Container mit einer oder mehreren offenen Abweichungen nicht für Schritte dieser Operation angemeldet werden. Dieses Feld ist nur als Bestandteil des separat lizenzierten Moduls "Nonconformance Management" verfügbar.

Optional

Durchsatzberichtsebene

Containerlevel, das von der Anwendung zur Berechnung und Erstellung von Berichten zum Durchsatz bei dieser Operation verwendet wird. Weitere Informationen finden Sie in der Beschreibung der Durchsatzzusammenfassung.

Erforderlich

Anzeigeoptionen

Definiert einen Satz von Anzeigeoptionen, die im Bereich zur Bildvisualisierung der Seite "Produktions-Client" zu verwenden sind.

Hinweis: Dieses Feld wird nur angezeigt, wenn Opcenter Execution Electronics installiert ist. Weitere Informationen finden Sie im Opcenter Execution Electronics-Benutzerhandbuch.

Optional

NPI-Dokumentenansicht

Gibt die NPI-Dokumentenansicht an, die standardmäßig im Visualisierungswerkzeug des Produktions-Clients angezeigt wird.

Hinweis: Dieses Feld wird nur angezeigt, wenn Opcenter Execution Electronics installiert ist. Weitere Informationen finden Sie im Opcenter Execution Electronics-Benutzerhandbuch.

Optional

Tabelle "Unzulässige Transaktion(en)"

Tabelle mit den Transaktionstypen, die für diese Operation nicht zulässig sind.

Optional

Vorgehensweise zum Definieren einer Operation

Gehen Sie folgendermaßen vor, um eine Operation zu definieren:

1. Öffnen Sie die Seite Operation. Die Seite "Operation" wird innerhalb der Seite "Modellierung" angezeigt.
2. Klicken Sie auf Neu. Es werden leere Felder zum Definieren einer neuen Instanz angezeigt.
3. Geben Sie einen Namen für die Operation in das Feld Operation ein.
4. Geben Sie eine Durchsatzberichtsebene ein.
5. Geben Sie abhängig von den jeweiligen Geschäftsanforderungen optionale Informationen ein. Informationen zu den optionalen Feldern finden Sie in Tabelle mit den Felddefinition.
6. Klicken Sie auf Speichern. Die Anwendung speichert das Modellierungsobjekt und zeigt eine Erfolgsmeldung an.

So wird eine Transaktion angegeben, die für diese Operation nicht zulässig ist

Gehen Sie folgendermaßen vor, um eine Transaktion anzugeben, die für diese Operation nicht zulässig ist:

1. Führen Sie die Schritte unter "Vorgehensweise zum Definieren einer Operation" aus.

Oder

Wählen Sie eine vorhandene Instanz einer Operation aus.

2. Erweitern Sie den Abschnitt Details.
3. Klicken Sie in der Tabelle "Unzulässige Transaktion(en)" auf Neue Zeile hinzufügen. Eine neue Zeile wird angezeigt.
4. Wählen Sie im Feld Transaktionstyp eine Transaktion zur Eingabe aus.
5. Wiederholen Sie ggf. die Schritte 3 und 4, um eine weitere unzulässige Transaktion hinzuzufügen.
6. Klicken Sie auf Speichern. Die Anwendung zeigt eine Erfolgsmeldung an, die angibt, dass das Modellierungsobjekt aktualisiert wurde.
