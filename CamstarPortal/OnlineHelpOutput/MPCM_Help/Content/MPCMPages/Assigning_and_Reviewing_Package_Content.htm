Assigning and Reviewing Package Content

You can assign modeling object instances and references to any existing package that is at a draft step or a step configured to allow assignment and review. The Assign and Review Content page enables you to customize and review the content of an existing package.

Before Assigning Package Content

Some modeling object instances available for selection on the Assign and Review Content page should not be assigned to a package for deployment to a target system. Some modeling objects may cause conflicts when activating packages on a target system.

Modeling Objects That Should Not Be Assigned to Package

Siemens recommends you do not assign these modeling object instances to a package for deployment to a target system.

Additionally, selecting Add References may automatically add these modeling objects to the Object Type Name grid. Siemens recommends you manually remove these instances from the grid to avoid conflicts.

Note: Adding modeling objects to the Modeling Objects to Exclude grid on the Factory modeling object ensures those instances do not appear for selection during export. Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information.

Modeling Object

Reason Not to Assign

Action

Action modeling object instances contain references to pages. Pages may contain different IDs from one system to another.

Data Collection Def

Data collection definitions require Designer and metadata updates.

Note: This does not apply to user data collection definitions.

Mfg Calendar

Overriding an existing calendar may have unintended consequences on history. Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for an alternative method to load a calendar from Excel.

Portal Menu Definition

Instance IDs are not likely to match from one system to another unless the database has been cloned. There is a potential to orphan menu items.

Role

Instance IDs are not likely to match from one system to another unless the database has been cloned.

UI Page Flow

You should export and import UI Page Flows using Portal Studio. Refer to the Opcenter Execution Medical Device and Diagnostics Portal Studio User Guide or the Opcenter Execution Core Portal Studio User Guide for information.

UI Virtual Page

You should export and import UI Virtual Pages using Portal Studio. Refer to the Opcenter Execution Medical Device and Diagnostics Portal Studio User Guide or the Opcenter Execution Core Portal Studio User Guide for information.

Web Part

You should export and import Web Parts using Portal Studio. Refer to the Opcenter Execution Medical Device and Diagnostics Portal Studio User Guide or the Opcenter Execution Core Portal Studio User Guide for information.

Modeling Objects That May Cause Conflicts

Siemens recommends you carefully consider assigning these modeling object instances to package for deployment to a target system.

Modeling Object

Reason Not to Assign

Employee

An employee modeling object instance may contain circular references. Instances may also reference role IDs that do not exist in the target system.

Modeling ESig Requirement

A Modeling ESignature requirement instance may contain circular references. The users, roles, and electronic signatures may not be consistent from source to target.

Numbering Rule

Conflicts may occur with numbering rule instances when the number on the source system is different from the number on the target system.

Organization

An organization modeling object instance may contain circular references.

History View

The manufacturing audit trail display may change unexpectedly on the target system. Custom changes on the target system may be overwritten.

Scheduled Business Rule

Importing a scheduled business rule exported from another system will update the start date on the target system. Additionally, you may need to restart the Camstar Deployment Server and the Camstar Notification Server if the scheduled business rule instance being updated or created on the target system affects deployment or notifications.

Revisioned Objects (ROs)

The application sets the first revision included in the import of revisioned objects as the Revision of Record when:

• No Revision of Record exists on the target system for an imported RO, and
• The export does not include a Revision of Record for the RO.

Deprecated Modeling Objects You Can Assign to a Package

Existing customers who use modeling object instances that have been deprecated in later releases can still export and import these modeling object instances:

• MSMQ Transport
• Scale
• Scale Group
• Scale Code
• Scale Status Reason
• Web Service Transport

When Assigning and Reviewing Package Content

Only package owners and collaborators can assign content to packages. The Assign and Review Content page displays one or two tabs depending on whether content collaboration has been enabled for the package:

• Assign and Review Content

The Assign and Review Content tab enables you to assign modeling object instances and references to a package. You can also display summary information for the package. You can also display summary information for the package and display all assigned objects and their references in a hierarchical view.

• Collaborator Tasks

The Collaborator Tasks tab appears only if content collaboration has been enabled for the package. The tab consists of the Collaborators grid and an instruction section. The Collaborators grid lists each collaborator's level, name, status of the collaboration work, dates on which the collaboration work was assigned, due, and completed, collaborator comments, and collaborator instructions. This information enables the package owner to view the status of the collaboration work and determine whether follow-up is needed. The instruction section lists the package owner's instructions for all collaborators and instructions specific to the selected task in the Collaborators grid. The instructions are display only. The instructions display only if you are a collaborator assigned to the package. Refer to "Updating a Package" for information on enabling content collaboration.

Note: The application does not display the instructions if you cannot perform a collaboration task. This occurs if the package owner or a user with the Owner Role assigns content and the Substitute Option is set to None for all tasks.

Only modeling objects for which instances have been defined are available for selection on the Assign and Review Content tab. The application does not display any modeling objects without defined instances.

You can search for a specific modeling object in the Objects list by entering the first few characters or the entire modeling object name in the filter field. The application displays all instances defined for a selected modeling object in the Instances list. You can also filter the list of instances. Refer to "Filtering the List of Instances" for information.

Note: Settings in the Designer application determine whether modeling object instances are available to add to a package. Non-exposed, abstract modeling objects are not available. Refer to the Opcenter Execution Medical Device and Diagnostics Designer User Guide or the Opcenter Execution Core Designer User Guide for information

You can select the Select All check box in the Instances header row to select all displayed instances automatically or you can select the check boxes for individual instances.

Clicking the Add to Grid button adds the selected instances to the Object Type Name grid and updates the Package Content Summary information. Entries in the grid appear in alphabetical order by modeling object name and then by modeling object instance name. Added instances remain in the Instances list but are shown as checked and disabled.

Note: All modeling object and where used instances should be added before references are added.

Adding references is optional when assigning package content. Objects are still deployed in the correct order if you choose not to include references. You must assign modeling object instances in the correct order or add references only when the Change Mgt Save option is set to the On position. Refer to "Assigning Modeling Object Instances to a Package in Modeling" for information on the Change Mgt Save option.

Example:

You select to assign the Batch, Rack, and Tray instances of the Container Level modeling object to the package. Clicking the Add to Grid button displays Container Level in the Object Type Name grid and updates the Pkg Content Summary table to indicate one modeling object and 3 instances. Additionally, Batch, Rack, and Tray remain in the Instances list, but they are checked and disabled.

Filtering the List of Instances

The application provides two methods for filtering the instances displayed in the Instances list:

• Entering the name of the instance you want to locate. The application updates the list of instances as you enter each character in the Instances filter field. The application displays only the instances containing those characters after you click the Refresh icon.
• Clicking the Filter icon to use criteria other than the instance name. The application displays the Instance Filters pop-up when you click the Filter icon. Use this pop-up to specify criteria such as timestamp, last employee to update the instance, and so on. Click Apply to close the Instance Filters pop-up and view all instances meeting these criteria.

The application displays an icon indicating the list has been filtered next to the Filter icon.

Clicking the Refresh icon removes all filtering criteria and the application displays all instances defined for the selected object. Selecting a different modeling object in the Select Objects field also removes all filtering criteria. Clicking the icon next to the Filter icon also resets the filter.

Viewing Where Instances Are Used

Clicking the Where Used button displays the Where Used pop-up when:

• You have selected one or more instances in the Object Type Name grid, and
• Change management settings have been specified for the factory associated with the logged in employee.

The Where Used pop-up displays the modeling object instances that directly reference the instances selected in the Object Type Name grid. For example, the application displays all Specs that reference the Operation instance you selected. The application does not display the Workflows that reference the Specs that reference the Operation instance you selected.

Note: Change management settings determine how many instances you can select in the Object Type Name grid. Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information on configuring Change Mgt Settings.

You can add the Where Used modeling object instances to the change management package. Select the instances in the Where Used pop-up and click Add.

Note: The application displays an error message when you click the Where Used button when any of these conditions are true:
• No change management settings are associated with the logged in user.
• No instances are selected in the Object Type Name grid.
• The number of instances selected exceeds the Where Used limit configured in the change management settings.

When Assigning a Large Number of Instances and References

The application displays a warning message if the content assigned to the package exceeds 1000 instances, including references. The message warns that system performance may be impacted, but the application does not prevent you from saving and later deploying the package.

Siemens recommends you review the Private Memory Usage application pool setting for the CamsarAppServer application pool. You may need to increase the value when working with 1000 or more instances. Refer to the Opcenter Execution Medical Device and Diagnostics System Administration Guide or the Opcenter Execution Core System Administration Guide for information on configuring the maximum memory limit for the application pool.

Note: The application displays the warning message only once even if you continue to exceed the 1000 instance threshold for the package.

The application may time out or produce unexpected results when exporting a large number of instances and references. You can increase timeout settings by updating values in the following locations:

• Management Studio - Select the Configuration node in the Sites tree and select the Database Connections tab. Increase the value in the Close Connections after being idle field to 500 seconds.
• Management Studio - Select the Configuration node in the Sites tree and select the Security tab. Increase the value in the Idle Session Timeout field to 30 minutes.
Note: Refer to the Opcenter Execution Medical Device and Diagnostics System Administration Guide or the Opcenter Execution Core System Administration Guide for information on Management Studio.
• XMLClientConfig.xml - The XMLClientConfig.xml file is located in C:\Program Files (x86)\Camstar\InSite Utility by default. Increase the values for SendTimeout and ReceiveTimeout to 500000.
• Web.config - The Web.config file is located in C:\Program Files (x86)\Camstar\CamstarPortal by default. Increase the executionTimeout value in the httpRuntime setting of the system.web node. The default value is 110 seconds.

The application enables you to add references to roles, but this may cause the application to time out when performing tasks on a package. You must remove references to roles from a package to complete these tasks.

When Completing Content Collaboration

You can view the status of the collaboration work in the Collaborators grid on the Collaborator Tasks tab. The application selects the default task for the currently logged-in collaborator when it loads the page. The instructions in the Instructions for Collaborator section contain instructions for the currently selected task. The application updates the instructions that display in this section if you select a different task in the grid. You can select only one task at a time in the grid.

You assign content to the selected package on the Assign and Review Content tab. If you are a collaborator, you click Complete on the Assign and Review Content tab when you finish performing your collaboration tasks.

Note: The collaborator can also assign content to or remove content from a package in Modeling. Refer to "Assigning Modeling Object Instances to a Package in Modeling" for information.

The Content Collaborator – Completed Assignment pop-up appears when you click Complete. This pop-up consists of the Collaborators grid and a comments section. The grid lists each collaborator's level, name, status of the collaboration tasks, and dates on which the collaboration tasks were assigned and are due.

The Collaborators grid lists only tasks that the currently logged-in collaborator is eligible to complete. Eligible tasks include:

• Tasks assigned to the currently logged-in collaborator at the current level or any prior levels.
• Tasks assigned to another collaborator at the current level or any prior levels only if the currently logged-in collaborator has the role defined with the Substitute Option.

The application selects the default task in the grid for the currently logged-in collaborator. You can select any task.

Any comments you enter in this pop-up appear in the Collaborator Comments field in the Collaborators grid on the Collaborator Tasks tab when you click Complete. The application also updates the Status and Completion Date fields on the Collaborator Tasks tab. The application sends an e-mail notification to the package owner to indicate that the collaborator has finished assigning content to the package if the package owner specified Completion Message to Owner on the Update Package page, Collaborators tab. It also sends an e-mail notification to the next collaborator (if specified) that his or her collaboration work can begin if the Notify Collaborators of Assignments on Update check box is selected on the Update Package page, Collaborators tab.

The application saves your latest comments to the Collaborator Comments field on the Collaborator Tasks tab if you click Complete multiple times.

The application sets Due Date to the Assigned Date and Status to Late on the Collaborator Tasks tab if the package owner did not specify Duration and Period when adding collaborators on the Update Package page.

The application enables collaborators to continue to assign content to the package after the Complete button has been clicked as long as the package is at a step at which assign and review content has been configured.

The application lists any new eligible tasks as you assign content and complete each task without requiring you to close the Assign and Review Content page after completing each task.

Viewing the Hierarchy of the Assigned Instances

Clicking the Hierarchy button displays the read-only Hierarchy pop-up. The Hierarchy pop-up displays an indented tree view of the modeling object instances added to the Object Type Name grid.

Instances added to the grid by clicking the Add to Grid button or by the Where Used pop-up appear in bold, black font. Instances added to the grid by clicking the Add References button appear in normal, black font under the bold entries.

You must add at least one modeling object instance to the Object Type Name grid to view the hierarchy. The application displays an error if you do not.

Viewing the Package Content Summary

Clicking the Pkg Content Summary icon below the Object Type Name grid displays the Package Content Summary pop-up. The Package Content Summary pop-up displays the following information:

• Number of modeling objects available for assignment
• Total number of modeling object instances available for assignment
• Number of modeling objects assigned to the package
• Total number of modeling object instances assigned to the package
• Number of modeling objects referenced by the selected modeling objects
• Total number of referenced modeling object instances assigned to the package

The application updates the Package Content Summary information as you add and remove instances and references to and from the package.

How to Assign Content to a Package

Follow these steps to assign content to the package:

1. Select Package Search from the Change Management menu. The Package Search page appears.
2. Enter or select one or more criteria to narrow your search.
3. Click Search. The application displays the packages matching your search criteria in the Search Results grid.
4. Select the package to which you want to assign content. The application displays the relevant transactions for the package.
5. Click Assign Content. The Assign and Review Content page appears.
6. Are you performing a collaboration task?

If . . .

Then . . .

Yes

Perform the "How to View Collaborator Tasks" procedure.

No

Go to step 7.
7. Select a modeling object from the Objects list. The application displays the available modeling object instances in the Instances list.
8. Do you want to filter the instances?

If . . .

Then . . .

Yes

Perform the "How to Filter the Instances" procedure.

No

Go to step 9.
9. Select the instances you want to assign to the package.
10. Click Add to Grid. The application adds the object and its selected instances to the Object Type Name grid and updates the Package Content Summary information.
11. Repeat steps 7-10 to add additional instances.
12. Do you want to add references to the package?

If . . .

Then . . .

Yes
a. Click the Add References button. The application adds all the referenced instances to the Object Type Name grid and updates the Package Content Summary information.
b. Go to step 13.

No

Go to step 13.
13. Click Assign. The application displays a success message.
14. Are you performing a collaboration task?

If . . .

Then . . .

Yes
a. Click Complete. The Content Collaborator – Completed Assignment pop-up appears.
b. Go to step 15.

No

The procedure is complete.
15. Do you want to complete the currently selected task?

If . . .

Then . . .

Yes

Enter text that describes the collaboration work you completed in the Comments field.

No
a. Select a different task in the Collaborators grid.
b. Enter text that describes the collaboration work you completed in the Comments field.
16. Click Complete. The application displays a success message, closes the Content Collaborator – Completed Assignment pop-up, updates the Comments, Status, and Completion Date fields on the Collaborator Tasks tab, updates the list of tasks with any new eligible tasks, and sends e-mail notification to the package owner and next collaborator if applicable.

How to Filter the Instances

Follow these steps to filter the instances:

1. Perform steps 1-7 of the "How to Assign Content to a Package" procedure.
2. Do you want to filter by name only or by multiple criteria?

If you want to filter by . . .

Then . . .

Name only

Enter the first few characters of the modeling object instance name in the Instances field. The application updates the list of instances to display only those instances containing those characters in the name.

Multiple criteria
a. Click the Filter icon. The application displays the Instance Filters pop-up.
b. Enter or select one or more criteria to narrow your search. Refer to the field definitions table for the Instance Filters pop-up for information about the fields.
c. Click Apply to filter the list and close the pop-up. The application updates the list of instances to display only those instances meeting your specified criteria.

How to View a Hierarchy of Assigned Instances

How to View Collaborator Tasks

Follow these steps to view collaborator tasks:

1. Perform steps 1-5 of the "How to Assign Content to a Package" procedure.
2. Click the Collaborator Tasks tab. The application displays all the collaboration tasks for the package and displays the instructions for the task currently selected in the Collaborators grid.
