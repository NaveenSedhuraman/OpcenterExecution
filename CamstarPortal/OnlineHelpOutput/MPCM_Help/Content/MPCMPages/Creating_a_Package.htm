Creating a Package

Creating a change management package is the first step in deploying updates to modeling data and process flows from a source system to one or more target systems.

Note: Teamcenter packages are created in CEP when a Teamcenter ECN is downloaded in to CEP. Refer to the "Teamcenter Change Management" appendix for information.

Before Creating a Package

You must configure change management in Modeling before you can create and deploy change management packages. You must define these modeling objects first:

• Package Creation Reason - Reasons for creating a package
• Server Catalog System - Source and target systems for a package
Note: The application displays an error message when you open the Create Package page if there are no server catalog systems defined.

You must assign roles with change management permissions to employees before creating packages. The employee creating a change management package must have a role with package creation permissions assigned. Siemens provides a Package Creator role with these permissions by default. Roles are assigned to employees in Modeling.

Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information on defining the modeling objects necessary to perform change management transactions. Refer to the Opcenter Execution Medical Device and Diagnostics System Administration Guide or the Opcenter Execution Core System Administration Guide for information on the default change management roles.

When Creating a Package

You must select a default or custom change management workflow when creating a package. The Information icon next to the Change Mgt Workflow field displays a diagram of the selected workflow in a pop-up when clicked. Additionally, the application displays a description of the change management workflow if one was provided when the instance was defined.

Note: You cannot update or change the assigned workflow after the package is created.

Siemens provides three workflows by default for creating and deploying a change management package:

• No Approval - Enables you to deploy change management packages from the source system without requiring approvals on the package.
• Camstar - Requires approvals on change management packages before deploying from the source system. Selecting this option enables the Approvers tab on the Update Package page, enabling you to view the approval template and the assigned approvers.
• PLM - Requires a Product Life Cycle Management (PLM) approval decision on change management packages before deploying from the source system.

Refer to "Understanding the Default Change Management Workflows" for information on the default workflows.

You can specify deployment information when you create the package or later when you update the package. You assign modeling and process flow information to the package later before deployment. You must select at least one target system before deploying a package.

The application uses information entered when performing the create package transaction to display core attributes of the package on successive change management package pages. The application displays the core attributes entered during package creation on the Update Package page. Core package attributes include information such as package name, step, and owner. Refer to "Package Information" for information on the core attributes and the pages on which they appear.

When Using a Package Creation Template

You can define a package creation template page using the Create Package Template modeling object. Selecting a template populates the Create Package page with the predefined package and deployment details. The applicable predefined approver and collaboration templates display on the Update Package page when the package is created. You do not have to enter all of the information manually each time you create a package.

You can overwrite the template information in the page. Selecting a different package creation template also overwrites the information in the page. The package information is not saved until you click Create or Create and Go to Update. Changes made upon package creation are only applied to the newly created package. The application does not save any updates you make to the package to the package creation template.

When Adding Deployment Details

Deployment details are optional when you create a package, but required before you can deploy a package.

The application displays a warning message if you select a target system that has a different MDB Version or application version defined in the associated Server Catalog System than from the source system's Server Catalog System. The warning message includes a warning for both the MDB Version and application version if both are different on the target and source systems. This warning message only displays for the mismatched target closest to the bottom of the grid. You can close the warning message and create the package. The application does not prevent you from creating a package if the MDB Versions or application version between source and target systems are different.

When Adding Activation Details

You can specify prerequisite packages to be activated prior to the activation of your package to account for dependencies on instances in other packages. Clicking Add on the Prerequisite Change Packages grid opens the Package Search pop-up. You can use the pop-up to search for and select prerequisite packages to add to your package.

When Creating a Package

Clicking the Create and Go to Update button creates the package and displays the Update Package page. You can modify the package's core attributes and assign approvers if the associated workflow requires approvals. Refer to "Updating a Package" for information.

Note: You must have the appropriate role to update a package.

Clicking the Create button creates the package and clears the fields on the Create Package page. You can begin entering information for a new package.

How to Create a Package

Follow these steps to create a package:

1. Select Create Package from the Change Management menu. The Create Package page appears.
2. Do you want to use a package creation template?

If . . .

Then . . .

Yes
a. Select a Package Creation Template. The application updates the Create Package page to show the predefined information associated with the selected template.
b. Go to step 3.

No

Go to step 3.
3. Enter a name for the change management package in the Package Name field.
4. Select a Package Owner.
5. Select a Package Creation Reason.
6. Select a change management workflow for package creation and deployment from the Change Management Workflow field.
7. Enter optional information according to your business requirements. Refer to the field definitions table for information about the optional fields.
8. Do one of the following:
• Click Create. The application creates the change management package and displays a success message.
• Click Create and Go to Update. The application creates the change management package and displays the Update Package page with the package information populated.
