Understanding Change Management Package Permissions

These permissions determine the actions the logged-in user can perform on a change management package:

• Page permissions
• Transaction permissions
• Allowable transaction permissions
• Change management workflow assigned to the package

Page, transaction, and allowable transaction permissions are assigned to roles. Roles are assigned to employees and to change management specs. Change management specs are assigned to change management workflows.

Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information on defining these modeling objects:

• Roles
• Employees
• Change management specs
• Change management workflows

Refer to the Opcenter Execution Medical Device and Diagnostics System Administration Guide or the Opcenter Execution Core System Administration Guide for information on change management security administration.

Page Permissions

The logged-in user must have permission to access the page where the transaction is to be performed. The page permission must be assigned to one of the roles assigned to the user. For example, the user responsible for creating a change management package must have the permission to access the Create Package page assigned to one of their roles.

Siemens provides a Default Pages role that has the StartChangePkg_VP Portal page permission assigned by default. You can add the page permission to any role.

Transaction Permissions

The logged-in user must have permission to access the page and to perform the transaction on the page. The transaction permission must be assigned to one or more of the roles assigned to the logged-in user. For example, the user responsible for creating a change management package must have the permission to perform the create (start) package transaction on the Create Package page.

The default Package Creator role has the Start Change Package transaction permission assigned by default (Start Change Package with Perform and Dispatch Override access modes). You can add the transaction permission to any role.

Allowable Transaction Permissions

The logged-in user must have permission to perform transactions on the change management package at its current step in the assigned workflow. The allowable transaction roles assigned to a change management spec and the permissions assigned to those roles determine the actions the logged-in user can perform on the package.

Note: Roles are associated with change management specs in the Allowable Transactions grid on the Change Mgt Spec modeling page.

Allowable transaction permissions are in addition to the page-level permissions and the page-level transaction permissions assigned to the logged-in user. The allowable transactions specified for each change management spec determine the pages on which the logged-in user can perform transactions.

For example, a user assigned the default Package Owner role can perform almost every action on a change management package. However, the default DraftPermissions role limits the logged-in user to performing these change management actions when a change management package is at the default draft step:

• Searching for a package
• Updating a package
• Assigning instances and dependencies to a package
• Altering the step of a package
• Deploying a package
• Viewing package details

The DraftPermissions role is assigned to the default Draft spec.

Workflow Permissions

The change management workflow assigned to the change management package determines the business process for the package. The step where the package is in the workflow, along with the page and transaction permissions, limit the transactions the logged-in user can perform on the package and the next steps for the package. For example, a package at the Draft step in the default No Approval workflow can be updated, viewed, assigned content, and deployed when:

• The logged-in user is assigned a role with package ownership permissions, and
• The role assigned to the Draft change management spec has package update, package details, assign and review content, and deployment permissions.

A user without package ownership permissions would not be able to update a package even though the role assigned to the Draft change management spec permits this transaction.

You can add transaction permissions to any role and assign the role to an employee and to any change management specs.

Understanding the Permissions Example

The X in the column indicates when Operator 1 can view and transact on the page. Each level of added permissions restricts the actions the logged-in user can perform:

• Operator 1 is assigned the Default Pages role, which specifies the pages a user can access. The Default Pages role allows access to all Portal pages.
• Operator 1 is assigned the Package Deployer role, which specifies the transactions a user can perform on the change management pages.
• The Draft step in the No Approval workflow is assigned the DraftPermissions role, which specifies the transactions the logged-in user can perform on a change management package at this step.
Note: The DraftPermissions assigned to the Draft spec in this scenario does not further limit the actions Operator 1 can perform on this package at this step.
