Understanding Approval Notifications

The application can send e-mails to employees notifying them a change management package is ready for their approval. Approval notifications are valid only for change management packages on workflows containing change management specs requiring approval.

The package owner assigns approvers to the change management package on the Update Package page. Refer to "Updating a Package" for information on assigning approvers to a package.

The application uses the employees listed in the Names column of the Approvers grid as the e-mail distribution list. Employees designated as package approvers receive e-mails when the package owner routes the package for approval.

Note: Approval notifications are not used in the default MOM PLM workflow for Teamcenter packages. Refer to the "Teamcenter Change Management" appendix for information on the Teamcenter package approval process.

Configuring Approval Notifications

A system administrator must configure some objects in Modeling specific to change management approvals before the package owner routes the package for approval. The objects that must be configured enable the application to send e-mail to specific employees. An optional object enables you to use a global setting for the e-mail notification.

Required Configuration

A system administrator must perform the following setup tasks in Modeling to ensure employees receive notifications:

• Define the SMTP transport to be used for e-mail notifications.
• Define or update a factory modeling object instance and select the SMTP transport you defined for e-mail notifications.
• Define or update employee modeling object instances to include the following:
• E-mail address where the employee will receive approval notifications
• Factory where the SMTP transport is associated
• Define e-mail messages for approvers using the E-mail Message modeling object.

Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information on defining and updating the SMTP Transport, Employee, E-mail Message, and Factory modeling objects.

Optional Configuration

A system administrator can perform the following when setting up e-mail notifications:

• Define e-mail notification types using the Change Mgt Settings modeling object. The Approvers grid on the Update Package page defaults to the e-mail option associated with the Change Mgt Approver notification type on the Change Mgt Settings modeling object.
Note: A system administrator can reassign a package approval task from one employee to another if, for example, the employee to which the task was originally assigned will be on vacation when the task is due. The administrator can select the Change Mgt Approval Delegation notification type on the Change Mgt Settings modeling object and select the e-mail message to be sent when the work is reassigned.
• Define package approval reminder notifications using the Change Mgt Settings modeling object. The application can send these reminders to the owner, approvers, or approver escalation recipients when a package approver has upcoming, due, or overdue approvals.
• Define or update a factory modeling object instance to include the global change management settings specified within the Change Mgt Settings object.

Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information on defining and updating the Change Mgt Settings modeling object.

Sending Approval E-Mail Notifications

The approval process starts when the package owner selects a package and clicks Route for Approval on the Package Search page. The application sends the approval notification e-mail to employees that have the lowest number specified for Level in the Approvers grid. The Approve (Camstar) button is available to these approvers on the Package Search page. The application sends the notification e-mail to only one employee if only one employee is designated.

Refer to "Routing a Package for Approval" for information.

These examples describe different approval scenarios.

Example 1: Multiple Approvers, Each at a Different Level

The following occurs:

• The application sends notifications sequentially.
• Each approver with a lower level number must approve the package before approvers with higher level numbers can begin.
• When the first approver approves the package, the application sends an e-mail notification to the approver with the next lowest number specified for Level.
• This process repeats until all designated approvers approve the package.
Note: The application moves the package to Rejected step if an approver rejects the package.

Example 2: Multiple Approvers at a Single Level

The following occurs:

• The application sends notifications in parallel.
• Approvers can approve the package at the same time.

Example 3: Multiple Approvers at Multiple Levels

The application can send the work and notifications as a combination of parallel and sequential. For example, if you set two approvers at Level 1, one approver at Level 2, and two approvers at Level 3, the following occurs:

• The application sends the approval notification to both approvers at Level 1 at the same time.
• The application sends the approval notification to the approver at Level 2 when both Level 1 approvers approve the package.
• The application sends notifications to both approvers at Level 3 when the Level 2 approver completes assigning content.
