Understanding Change Control Enforcement

Siemens provides the ability to enforce change control on all or a subset of Modeling objects included in change management packages. There are two types of change control enforcement:

• Enforcement for instances in a package on any workflow - You can assign a modeling object instance to a package associated with any change management workflow.
• Enforcement for instances in a package on a workflow with an approval step - You can assign a modeling object instance to a package associated with a change management workflow containing an approval step.

The application allows only one type of change control enforcement per modeling object.

When Setting Up Change Control Enforcement

You must define a master data catalog to enforce change control. The master data catalog enables you to specify:

• The modeling object instances required to be included in a change package, and
• Whether the instances can be included with any change package or only those packages with an approval step.

You must select the Requires Approval check box on a minimum of one change management spec in an approval change management workflow. The application uses the Requires Approval check box with the master data catalog to enforce change control for those modeling object instances that should only be associated with a package on an approval workflow.

The application uses the master data catalog and the Requires Approval check box to ensure modeling object instances are assigned to change packages on the appropriate workflows. Refer to the Opcenter Execution Medical Device and Diagnostics Modeling User Guide or the Opcenter Execution Core Modeling User Guide for information on defining master data catalogs and change management specs.

The application displays an error message when you save a modeling object instance for which change control has been enforced and:

• You have not associated the instance with a package.

Or

• You assigned the instance to the wrong workflow (for example, the modeling object instance must be assigned to a workflow containing an approval step and you assigned it to a non-approval workflow).

Change Control Enforcement Examples

These examples describe different change control enforcement scenarios.

Example 1: 

No change package exists. Modeling works as it always has.

Example 2:

A change package exists and is associated with a change management workflow, but no master data catalog has been defined:

• The application cannot enforce change control.
• You can define any new modeling object instances without the application requiring you to add them to the package.
• You can assign any modeling object instances to the package regardless of the associated workflow type (approval or non-approval).

Example 3:

A change package exists and is associated with a change management workflow, and a master data catalog has been defined. The application:

• Enforces change control.
• Requires you to assign modeling object instances to a package if the master data catalog indicates change control should be enforced for those instances.
• Uses settings in the master data catalog and the Requires Approval check box setting on change management specs within the associated workflow to ensure the modeling object instance is added to the package.

For example, the master data catalog specifies change control enforcement with approval for the Factory modeling object. The application evaluates the specs within the workflow associated with the package to which you are trying to add the Factory instance. The application allows you to add the Factory instance to the package if the Requires Approval check box has been selected on at least one spec in the associated workflow. The application displays an error message if the Requires Approval check box has not been selected on at least one spec in the associated workflow.
